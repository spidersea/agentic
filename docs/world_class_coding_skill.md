# 通用代码开发与执行技能 (World-Class Agentic Coding Skill)

> 本技能基于"How To Be A World-Class Agentic Engineer"原则提炼，适用于所有项目的代码研发。它致力于解决大模型"上下文过载"、"谄媚幻觉"和"边界模糊"的问题，最大化智能体的编码与执行效能。
>
> **根本心法**: 不要追逐花哨的外部插件和工具链。真正有用的能力会被前沿公司内置到产品中。保持工具链精简，只用基础 CLI + 理解核心原则，就足以完成世界级的工作。

## 核心原则 (Core Principles)

1. **少即是多 (Minimal Context)**: 将"研究/发散"与"执行/编码"严格分离。切忌将不相关的历史上下文和实现方案带入最终的编码执行阶段。
2. **精确指令 (Precise Instructions)**: 绝不下达"去搭个XX系统"这类含糊指令。所有任务指令必须精确到技术选型、库版本和核心逻辑流。如果连自己都说不清细节，先开一个研究会话去定型。
3. **消除谄媚 (Neutral Validation)**: 在检查代码和 debug 时，使用中立陈述（如"遍历代码逻辑并汇报观察结果"），而不是预设前提（如"帮我找出bug"）。
4. **终点契约 (Task Contract)**: 任务有明确的终结边界，不达标不可自行宣布完成。
5. **会话隔离 (Session Isolation)**: 每个独立任务/契约应在新的会话中执行。禁止在一个长会话中串联多个不相关任务，防止跨任务上下文污染。
6. **规则卫生 (Rule Hygiene)**: 定期清理和合并 Rules、Skills 及 AGENT.md 中的条目，移除过时或互相矛盾的规则。规则膨胀会导致智能体性能退化。
7. **人类兜底 (Own The Outcome)**: 没有任何 Agent 是完美的。可以将大量设计和实现工作委托给 Agent，但最终的质量责任始终由人类承担。必须审查关键产出，不可盲信。

## 反模式清单 (Anti-Patterns)

> 以下做法将严重降低智能体效能，必须坚决避免。

- ❌ 一次性给 Agent 加载超过 5 个与当前任务无关的文件
- ❌ 使用预设偏见的 debug 提示词，例如：
  - "帮我找出这个 bug" → Agent 会**制造**一个 bug 来满足你
  - "这段代码有什么问题" → Agent 会强行挑刺，即使代码是正确的
- ❌ 在一个会话中串联超过 3 个不相关任务（上下文污染）
- ❌ 在 AGENT.md 中堆积上百条未分类的规则（规则膨胀）
- ❌ 给 Agent 需求描述时只说"搭个XX系统"而不给技术细节（模糊指令）
- ❌ 上下文被压缩/重置后，不重新读取任务计划就继续编码（记忆残影）

## 实施标准流程 (Standard Operating Procedure)

### Phase 1: 调研与技术定调 (Research & Scoping) → PLANNING 模式

- **触发条件**: 当需求较为空泛或具体技术细节未敲定时。
- **动作**: 
  - 开辟独立的**研究会话/上下文区域**，不对具体代码进行修改，仅收集实现途径和对比优劣。
  - 产出**精确的技术规格说明**（如："使用 React + Vanilla CSS 实现 UI，路由采用 React Router，交互状态使用 Zustand"等）。
- **完成标志**: 得到一份极度精确、无歧义的技术实施清单。

### Phase 2: 确立验收契约 (Establish Task Contract) → PLANNING 模式

- **触发条件**: 准备进入实质性编码阶段前。
- **动作**:
  - 创建 `{TaskName}_CONTRACT.md` 或在备忘录中明确列出验收清单。
  - 清单必须包含：
    - **测试用例**: 代码写完后必须通过的自动化或手工验证节点。
    - **视觉断言**: 若为 UI，说明截屏需达到的精确表现形态。
    - **行为断言**: 组件交互时的状态流转结果。
- **强制规则**: 除非所有契约条件跑通（全绿测试/UI吻合），否则任务（Session）**绝不可标记完成（Done）**。

### Phase 3: 严格上下文编码 (Context-Strict Implementation) → EXECUTION 模式

- **触发条件**: 技术规格与契约已拟妥，正式编写代码。
- **动作**:
  - 清理多余记忆流（Flush Context Bloat），重置大脑。
  - **按需加载**: 只允许读取本次开发最直接关联的少数文件。读取前务必再扫一眼"计划纲领"和"契约"。
  - 依样画葫芦，百分百遵循 Phase 1 确定的精确技术栈进行敲击。不自行衍生未经授权的第三方依赖或"自作主张的脑补常识"。
- **上下文恢复协议 (Compaction Recovery)**: 每次上下文被压缩/重置后，**禁止凭"记忆残影"继续工作**，必须重新读取：
  1. 任务计划纲领
  2. 验收契约 `{TaskName}_CONTRACT.md`
  3. 当前正在修改的核心文件
  
  确认上下文完整后，再继续编码。

### Phase 4: 中立验证与对抗验收 (Verification & Adversarial Refereeing) → VERIFICATION 模式

- **触发条件**: 初版代码完成，开始执行契约验收。
- **动作**:
  - 运行自动化测试或进行功能模拟。
  - **Review 代码**: 采用"中立型"提示词，不带预设偏见。
  
    | ❌ 有偏见的提示 | ✅ 中立的提示 |
    |---|---|
    | "帮我找出这段代码的 bug" | "逐步跟随这段代码的执行逻辑，详述实际执行流并汇报观察结果" |
    | "这个函数有什么问题" | "阅读这个函数的输入、处理、输出，描述它实际做了什么" |
    | "为什么数据库查询这么慢" | "分析这段数据库查询的执行计划，汇报每一步的耗时和数据量" |

  - **深度 Debug (对抗模式)**: 若遇到复杂漏洞，启用左右互搏——
    - 专家A（极度敏锐）：尽可能找出所有可能的风险点和代码瑕疵（包括误报）。
    - 辩手B（极度严格）：逐条尝试推翻专家A寻找的风险点，证明业务逻辑是可靠的。
    - 裁判官C（掌握真相）：根据客观的日志/运行结果，裁定 A 和 B 的辩论，得出最真实的 Bug 列表。
  - 修复真正被验证的缺陷后，若完整契约依然通关，则宣告核心任务彻底闭环。

## Rules 与 Skills 的区分

在 `AGENT.md` 中管理的配置分为两类，切勿混淆：

| 类型 | 定义 | 用途 | 示例 |
|---|---|---|---|
| **Rules（规则）** | 编码偏好与禁令 | 告诉 Agent **不要做什么** 或 **偏好什么** | "禁止使用 `any` 类型"、"日志必须用结构化格式" |
| **Skills（技能）** | 操作配方与流程 | 告诉 Agent **怎样做某件事** | 本文档、部署流程、测试编写规范 |

> **提前预审技巧**: 如果你不确定 Agent 会如何解决一个问题，让它先研究并**写成 Skill**。你可以在 Agent 实际执行前审查和修正它的解题思路。

## 如何在你的项目中集成与调用？

1. **统一入口**: 将系统引流总配置写入项目根目录下的 `CLAUDE.md` / `AGENT.md` 中。把 `AGENT.md` 当做一个**逻辑路由表**，而不是规则大全——它只负责告诉 Agent "在什么场景下，去读哪个文件"。
2. **分发路由**: 在 `AGENT.md` 里用条件逻辑组织上下文加载，例如：
   ```
   - 当需要开发新功能 → 读取 `skills/world_class_coding_skill.md`
   - 当需要写测试 → 读取 `rules/testing-rules.md`
   - 当测试失败 → 读取 `rules/test-failing-rules.md`
   ```
3. **保持环境整洁**: 定期（建议每 2-4 周）清理 `AGENT.md`、Rules 和 Skills 文件，移除过时或互相矛盾的条目。确保每次调用本技能时，Agent 的大脑都是清醒且无额外污染的。若规则超过 20 条或 Agent 需要读取超过 5 个 markdown 文件才能开工，说明已经到了必须清理的时刻。
4. **从简开始，逐步迭代**: 不要一开始就堆砌大量规则。像培养新助手一样——从零开始，在实际使用中逐步发现 Agent 的不良行为，每次只新增一条针对性规则。当规则之间开始出现矛盾或 Agent 性能退化时，进行一次全面清理合并。
