# 自动化测试流程

> 本流程遵循 SKILL.md 第七章的自动化测试纪律，确保测试的质量和覆盖率。
> 触发方式: `/test`

## 步骤

1. **确认测试范围**
   - 向用户确认需要测试的模块或功能
   - 读取相关代码文件和现有测试文件
   - 📊 如果代码图谱可用，查询已有测试覆盖：
     - `query_graph(pattern="tests_for", target="<目标函数/类>")` — 避免重复编写已有测试
     - `query_graph(pattern="children_of", target="<目标文件>")` — 了解文件中有哪些函数需要覆盖
     - 对比已有测试和目标函数列表，识别覆盖缺口
   - **覆盖率目标**: 核心业务逻辑 ≥ 80%；关键路径（auth、支付、数据写入）≥ 95%

2. **确认测试基础设施**
   - 确认项目使用的测试框架（如 pytest、jest、mocha 等）
   - 确认测试运行命令（如 `npm test`、`pytest -v`）
   - 确认是否有现有的测试配置文件

3. **按测试金字塔编写测试**
   - **单元测试**（必须）:
     - 每个核心函数至少一个正常路径 + 一个异常路径
     - 命名格式: `test_<功能>_<场景>_<预期结果>`
     - 使用 mock/stub 隔离外部依赖
   - **集成测试**（推荐）:
     - 模块间有交互时编写
     - 验证数据流和状态流转
   - **E2E 测试**（视项目需要）:
     - 关键用户流程编写
     - 验证完整业务路径

4. **测试质量检查**
   - 每个测试只验证一个行为（Single Assertion Principle）
   - 测试之间相互独立，不依赖执行顺序
   - 禁止在测试中使用 `sleep` 或硬编码延迟
   - 测试应当能独立运行，不依赖特定环境状态

4.5 **测试有效性审计（防止「测了但没测到点子」）**
   完成测试编写后，必须自我审计以下 5 项，并在结果报告中逐条回答：
   - ✅ **是否测了真实行为**（而非仅验证「函数被调用了」）？举例：`assert result == expected` 而非 `assert mock.called`
   - ✅ **是否有失败路径测试**（错误输入、异常情况、边界值）？每个公共函数至少一个异常路径
   - ✅ **Mock 占比是否合理**（不超过测试总量的 50%）？过度 mock 等于没有真正测试集成行为
   - ✅ **关键用户流程是否有 E2E / 集成测试覆盖**（不能全靠单元测试替代）？
   - ✅ **测试是否会在代码出 bug 时真正失败**（可通过临时改错逻辑来验证）？
   
   ⚠️ 如果上述任一项回答为「否」，必须补充对应测试再提交，不可跳过。

// turbo
5. **执行测试**
   - 运行全量测试套件
   - 记录测试结果（通过数 / 失败数 / 跳过数）
   - **失败处理流程**:
     - 分析失败原因，修复代码（不修改测试）
     - 重新运行，确认修复未引入回归
     - 若连续 3 次修复仍失败 → 停止并向用户上报，附完整失败日志
   - **失败禁止**: 不可将测试标记为 `skip` 或注释掉来绕过失败

6. **汇报结果**
   - 向用户汇报测试结果:
     ```
     测试结果: X/X 通过, 0 失败
     测试命令: [实际运行的命令]
     覆盖范围: [已覆盖的功能/模块列表]
     覆盖率: [核心模块 X%, 全量 X%]（如工具支持）
     未覆盖: [如有, 列出未覆盖的关键路径]
     ```
