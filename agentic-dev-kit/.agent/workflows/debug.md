# 中立 Debug 流程

> 本流程强制使用中立提示词和对抗验收模式，防止 AI 谄媚式制造不存在的 bug。
> 触发方式: `/debug`

## 步骤

0. **建立终点契约 (Task Contract)**
   > ⛔ **核心防御机制：对抗注意力衰减与产物断裂**
   - 收到 `/debug` 的**第一秒钟**，你必须立即使用写文件工具，在当前工作区创建一个名为 `debug_checklist.md` 的实体文件（可以写在根目录或 `.agent/state/` 下，或者整合进你的 `task.md` 中），强制列出本工作流的 9 个大步骤作为 `[ ] TODO` 打卡项。
   - **终点拦截规则：** 这是一个强状态契约。除非你亲手将该磁盘文件中的 9 个检查项全部标记为 `[x]`，否则**绝对不可**自行宣告问题已修复、终止会话或跳出流程。每完成下面的一步，你必须去物理更新该文件。一旦局部计划覆盖了它，你必须强行找回并继续执行后续阶段。

1. **读取开发规范**
   读取 `.agent/skills/world_class_coding/SKILL.md` Phase 0 + Phase 4。

1.5 **复杂度评估（内部，不问用户）**
   - **🟢 轻量**（用户明确指出了具体错误位置/文件）→ 跳过步骤 5（对抗验证），直接：中立分析 → 修复 → 跑测试
   - **🟡 标准**（知道症状但不知道原因）→ 执行步骤 2-8，跳过步骤 5
   - **🔴 完整**（复杂/跨模块/安全相关 bug）→ 执行全部步骤（含对抗验证）

2. **收集问题现象**
   - 🟢 **轻量模式**：如果用户已描述了具体错误（文件、行号、错误类型），跳过此步直接到步骤 3
   - 🟡🔴 **标准/完整模式**：
     - 向用户确认：具体的错误现象是什么？预期行为是什么？
     - 收集相关的错误日志、截图或复现步骤
     - 明确问题的范围（哪个模块/文件/功能）

   > ⛔ **铁律：NO FIXES WITHOUT ROOT CAUSE**
   > 未完成根因分析（步骤 3-4）前，**禁止提出任何修复方案**。
   
   以下想法是**危险信号**，一旦出现必须停下回到步骤 3：
   - "先快速修一下，以后再查根因"
   - "试试改一下 X 看看行不行"
   - "应该是 X 的问题，让我修一下"
   - "已经改了 2 次了，再试一次"
   
   | 常见借口 | 现实 |
   |---|---|
   | "问题很简单不需要流程" | 简单问题也有根因，流程对简单问题也很快 |
   | "紧急没时间走流程" | 系统性调试比瞎猜快（15-30min vs 2-3h） |
   | "先修了再查" | 第一次修复定下模式，一开始就要做对 |
   | "同时改多处节省时间" | 无法隔离哪个改动有效，还会引入新 bug |
   | "我看到问题了，让我修" | 看到症状 ≠ 理解根因 |

3. **中立代码审查**
   - ⚠️ **禁止使用偏见提示**，必须使用以下中立方式：
     - ❌ "帮我找出这段代码的 bug"
     - ✅ "逐步跟随这段代码的执行逻辑，详述实际执行流并汇报观察结果"
   - 📊 如果代码图谱可用，辅助定位问题范围：
     - `query_graph(pattern="callers_of", target="<出问题的函数>")` — 查上游调用方
     - `query_graph(pattern="callees_of", target="<出问题的函数>")` — 查下游被调用者
     - `query_graph(pattern="tests_for", target="<出问题的函数>")` — 查关联测试
     - 将调用链作为中立分析的辅助证据
   - 只加载与问题直接相关的文件（≤5 个）
   - 逐步跟踪代码执行逻辑，记录每一步的实际行为

// turbo
4. **对比分析**
   - 将实际执行流与预期行为逐项对比
   - 记录所有偏差点（不预判哪个是 bug）

5. **对抗验证（如需要深度分析）**
   - **专家 A**: 列出所有可疑偏差点，评估每个点的风险等级
   - **辩手 B**: 为每个可疑点寻找合理解释，尝试证明其正确性
   - **裁判 C**: 基于日志/运行结果/文件内容，裁定真实问题列表

6. **修复与验证（Escalation 对齐）**
   - **⚠️ 范围声明**：在开始修复前，明确本次调试的范围边界（例如：「仅修改 `src/auth/` 目录」），未经用户确认**禁止**修改范围外的文件
   - 仅修复被裁判 C 确认的真实问题
   - 运行相关测试验证修复是否生效
   - 修复失败时，按 `escalation/SKILL.md` 压力等级递进响应：
     - **L1（第 2 次失败）**: 切换**本质不同**的修复方案（换参数不算）
     - **L2（第 3 次失败）**: 搜索 + 读源码上下文 50 行 + 列 3 个新假设；指引切换方法论（参见 `autoresearch/references/methodology-router.md`）
     - **L3（第 4 次失败）**: 完成七项检查清单（`escalation/SKILL.md`），质疑架构层：这个 bug 是否说明当前架构/抽象本身有问题？
     - **L4（第 5 次+ 失败）**: 向用户汇报已尝试的全部方案，强制切换方法论。请求人类决策：继续 patch 还是架构改进
   - 确认修复未引入新问题（回归测试）

7. **举一反三 — 同类问题扫描（必须执行）**
   修复完当前 bug 后，**禁止直接结束**。必须执行以下扩展检查：

   **7.1 根因分析**
   - 这个 bug 的根本原因是什么？（而不仅仅是表面症状）
   - 是代码逻辑错误、边界遗漏、类型不匹配、竞态条件、还是设计缺陷？
   - 用一句话概括这个 bug 的**模式**（如"异步回调中未处理 null 返回值"）

   **7.2 同模式搜索（图谱增强）**
   - 📊 如果代码图谱可用：
     - `semantic_search_nodes(query="<bug模式描述>")` — 语义搜索同类代码
     - `query_graph(pattern="callers_of", target="<修复的函数>")` — 检查所有调用方是否有同类问题
     - 结合图谱结果和 grep 搜索，确保不遗漏同类问题
   - 如果图谱不可用，用传统 grep 搜索同类代码模式：
     - 相同函数/方法的其他调用点
     - 相同设计模式的其他实现
     - 同一作者/同一时期的类似代码
   - 对每个匹配点，用中立方式检查是否存在同类问题

   **7.3 修复范围扩展**
   - 如果发现同类问题，一并修复
   - 如果影响面较大，向用户汇报并指引是否在本次一并处理
   - 为同类问题补充自动化测试，防止复发

   **7.4 防御性措施（可选但推荐）**
   - 如果这类 bug 可以通过规则预防，指引添加一条规则到 AGENT.md
   - 如果这类 bug 可以通过测试捕获，补充回归测试
   - 如果这类 bug 源于架构问题，记录到技术债务清单

8. **回测验证 — 自动执行 `/test` 流程**
   修复和举一反三完成后，**必须执行回测**：
   - 按 `.agent/workflows/test.md` 的流程运行全量测试
   - 按测试金字塔顺序执行：单元测试 → 集成测试 → E2E 测试
   - 如果有任何测试失败：**回到步骤 6**，不可跳过
   - 全部通过后，向用户汇报回测结果

9. **记录结论**
   - 总结问题根因（不是症状）、修复方案、验证结果
   - 总结同模式搜索的结果（检查了哪些地方、发现了几个同类问题）
   - 记录回测结果（X/X 通过, 0 失败）
   - 如果产出了防御性规则或测试，一并记录
   - 如果发现了需要后续关注的风险，记录到任务备忘中
   - **检查终点契约**：确认已将物理文件 `debug_checklist.md` / `task.md` 中的所有 [ ] 转化为 [x]，方可宣告 `/debug` 流程完结。
   
   **🔄 主动改进指引**：
    - 同类问题 ≥3 个 → 指引 `/autoresearch:fix` 批量修复
    - 根因为架构缺陷 → 指引 `/autoresearch` 进行结构性重构
    - 修复涉及 ≥5 个文件 → 指引 `/autoresearch:security` 安全扫描受影响模块
