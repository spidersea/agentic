# 新功能开发流程

> 本流程严格遵循 `.agent/skills/world_class_coding/SKILL.md` 中的四阶段 SOP。
> 触发方式: `/new-feature`
>
> **工作流链接**: 本流程会自动调用 `/checkpoint`、`/test`、`/review`，未完成时可链接 `/handoff` → `/resume`。

## 步骤

1. **读取开发规范**
   读取 `.agent/skills/world_class_coding/SKILL.md`，理解四阶段 SOP 和核心原则。

1.5 **创建隔离工作区（推荐用于大型项目）**
   - 如果项目使用 Git 且修改范围较大（预计跨 5+ 文件）：
     - 使用 `git worktree add ../<项目名>-<功能名> -b feature/<功能名>` 创建独立工作目录
     - 在新分支上开发，避免污染主分支
     - 完成后通过 PR 合并
   - 对于小改动可跳过此步骤

2. **Phase 1: 调研与技术定调**
   - 与用户确认需求范围和边界
   - ⚠️ **每次只问一个问题**，不要用问题列表轰炸用户
   - 优先使用选择题（"方案 A 还是 B？"），必要时再用开放式提问
   - **必须提出 2-3 个替代方案并附带权衡分析和推荐理由**，不可只提供唯一方案
   - 研究现有代码库，识别相关模块和依赖
     - 📊 如果代码图谱可用，执行以下查询辅助调研：
       - `query_graph(pattern="file_summary", target="<目标模块>")` — 了解模块结构
       - `query_graph(pattern="importers_of", target="<目标文件>")` — 了解谁依赖它
       - `query_graph(pattern="callers_of", target="<核心函数>")` — 了解调用关系
     - 如果图谱不可用，用 grep/find 手动识别依赖关系
   - 产出精确的技术规格说明，必须包含：
     - 技术选型、版本、核心逻辑流、模块依赖、已知约束
     - **文件清单**: 每个将创建/修改的精确文件路径
     - **任务分解**: 将工作拆分为 2-5 分钟的小任务，每个任务包含精确文件路径、关键代码片段、验证命令和预期输出
     - **DRY / YAGNI 检查**: 每个功能点自问“这真的需要吗？有没有更简单的方式？”
   - 将技术规格以实施计划的形式提交用户确认
   - ⛳ CP-1: 执行 `/checkpoint` 记录技术规格状态

3. **Phase 2: 确立验收契约**
   - 基于技术规格，创建验收清单，包含:
     - 自动化测试用例（必须）
     - 视觉断言（若为 UI，必须）
     - 行为断言（状态流转验证）
     - 性能基线（如适用）
   - 验收清单提交用户确认
   - ⛳ CP-2: 执行 `/checkpoint` 记录验收契约状态

4. **Phase 3: 严格上下文编码**
   - 📊 如果代码图谱可用，在开始编码前：
     - 执行 `get_impact_radius(changed_files=[本次将修改的文件列表])`
     - 根据影响半径确定最多加载哪 5 个文件（优先级：变更文件 > 直接调用方 > 测试文件）
     - 如果影响半径 > 20 个节点，向用户预警修改范围可能过大，建议拆分
   - 清理无关上下文，只加载规格中定义的相关文件（≤5 个）
   - 严格按照 Phase 1 的技术规格编码，不引入未授权依赖
   - 每完成一个逻辑模块，立即执行 `/test` 运行该模块测试并确认输出（禁止未验证就声称“测试通过”）
   - ⛳ CP-3: 执行 `/checkpoint` 记录各模块编码和测试状态

// turbo
5. **Phase 4: 中立验证与验收**
   - 执行 `/test` 运行全量自动化测试
   - 执行 `/review` 对抗式代码审查（如有复杂逻辑）
   - 如发现缺陷，执行 `/debug` 修复（含举一反三 + 回测）
   - ⛳ CP-4: 执行 `/checkpoint` 记录最终验证状态

6. **工作流闭环验收**
   按以下闸门序列逐项验证，全部通过方可标记任务完成：
   - 🚪 **测试闸门**: 运行全量测试套件，确认 0 failures
   - 🚪 **契约核验**: 逐条核对验收清单中的每个标准
   - 🚪 **回归确认**: 确认本次修改未破坏已有功能
   - 🚪 **变更摘要**: 生成本次变更的文件清单和关键改动说明
   - 🚪 **人类审批**: 将闭环报告提交用户确认
   - 如果某步持续失败超过 3 次，暂停并上报用户
   - 如果有后续工作，使用 `/handoff` 生成交接备忘录
