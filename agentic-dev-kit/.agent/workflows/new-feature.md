# 新功能开发流程

> 本流程严格遵循 `.agent/skills/world_class_coding/SKILL.md` 中的四阶段 SOP。
> 触发方式: `/new-feature`
>
> **工作流链接**: 本流程会自动调用 `/checkpoint`、`/test`、`/review`，未完成时可链接 `/handoff` → `/resume`。

## 步骤

1. **读取开发规范**
   读取 `.agent/skills/world_class_coding/SKILL.md`，理解四阶段 SOP 和 Phase 0 复杂度路由。

1.5 **Phase 0: 复杂度评估（内部，不问用户）**
   按 `world_class_coding/SKILL.md` Phase 0 评估复杂度，选择执行档位：
   - **🟢 轻量**（≤2 文件、需求明确、无 API 变更）→ 跳到步骤 4（直接编码）
   - **🟡 标准**（3-7 文件、方向清楚）→ 执行简化版步骤 2-3
   - **🔴 完整**（8+ 文件、需求模糊、架构变更）→ 执行完整步骤 2-6
   
   如果项目使用 Git 且修改范围较大（预计跨 5+ 文件）：
   - 使用 `git worktree add ../<项目名>-<功能名> -b feature/<功能名>` 创建独立工作目录

2. **Phase 1: 调研与技术定调**
   
   **🟢 轻量模式 → 跳过此步骤，直接到步骤 4**
   
   **🟡 标准模式：**
   - Agent 自行研究代码库，识别相关模块和依赖
   - 产出简明方案：推荐方案 + 理由 + 影响范围
   - **只问 1 个关键决策**（如"方案 A vs B，推荐 A 因为 X"）
   - 用户确认后直接进入步骤 3
   - ⛳ CP-1: 可选
   
   **🔴 完整模式：**
   - 与用户确认需求范围和边界
   - ⚠️ **每次只问一个问题**，不要用问题列表轰炸用户
   - 优先使用选择题（"方案 A 还是 B？"），必要时再用开放式提问
   - **前提挑战（必须执行）**：接受需求前，主动挑战 1-2 个前提假设
   - **必须提出 2-3 个替代方案并附带权衡分析和推荐理由**
   - 研究现有代码库，识别相关模块和依赖
      - 📊 如果代码图谱可用，执行查询辅助调研
      - 如果图谱不可用，用 grep/find 手动识别依赖关系
   - 产出精确的技术规格说明（文件清单 + 任务分解 + 完整代码 + 运行命令）
   - 📋 **规格驱动（完整模式强制）**：执行 `/spec:propose` 生成结构化变更产物
   - 📄 **设计文件固化**：`git add openspec/ && git commit`
   - 将技术规格提交用户确认
   - ⛳ CP-1: 必须

3. **Phase 2: 确立验收契约**
   
   **🟢 轻量模式 → 跳过（改完跑测试通过即可）**
   
   **🟡 标准模式：**
   - 验收标准直接内联在 Phase 1 方案中（不单独创建文件）
   - 用户在确认方案时一并确认验收标准（不额外增加确认轮次）
   
   **🔴 完整模式：**
   - 基于技术规格，创建验收清单，包含:
     - 自动化测试用例（必须）
     - 视觉断言（若为 UI，必须）
     - 行为断言（状态流转验证）
     - 性能基线（如适用）
   - 验收清单提交用户确认
   - ⛳ CP-2: 执行 `/checkpoint` 记录验收契约状态

4. **Phase 3: 严格上下文编码**
   - 📊 如果代码图谱可用，在开始编码前：
     - 执行 `get_impact_radius(changed_files=[本次将修改的文件列表])`
     - 根据影响半径确定最多加载哪 5 个文件（优先级：变更文件 > 直接调用方 > 测试文件）
     - 如果影响半径 > 20 个节点，向用户预警修改范围可能过大，建议拆分
   - 清理无关上下文，只加载规格中定义的相关文件（≤5 个）
   - 严格按照 Phase 1 的技术规格编码，不引入未授权依赖
   - 每完成一个逻辑模块，立即执行 `/test` 运行该模块测试并确认输出（禁止未验证就声称"测试通过"）
   - 📝 **文档同步检查**：修改公共函数/类/接口后，执行 `grep -r "<函数名>" docs/ README.md *.md` 搜索所有引用该接口的文档（README、API 文档、注释、CHANGELOG），确认同步更新
   - ⛳ CP-3: 执行 `/checkpoint` 记录各模块编码和测试状态

// turbo
5. **Phase 4: 中立验证与验收**
   - 执行 `/test` 运行全量自动化测试
   - 执行 `/review` 对抗式代码审查（如有复杂逻辑）
   - 如发现缺陷，执行 `/debug` 修复（含举一反三 + 回测）
   - ⛳ CP-4: 执行 `/checkpoint` 记录最终验证状态

6. **工作流闭环验收**
   按以下闸门序列逐项验证，全部通过方可标记任务完成：
    - 🚪 **测试闸门**: 运行全量测试套件，确认 0 failures
    - 🚪 **契约核验**: 逐条核对验收清单中的每个标准
    - 🚪 **回归确认**: 确认本次修改未破坏已有功能
    - 🚪 **变更摘要**: 生成本次变更的文件清单和关键改动说明
    - 🚪 **人类审批**: 将闭环报告提交用户确认
    - 如果某步持续失败超过 3 次，暂停并上报用户

7. **分支完成**
    - 执行 `/finish` 完成分支收尾（merge/PR/keep/discard + worktree 清理）
    - 如果有后续工作，使用 `/handoff` 生成交接备忘录
