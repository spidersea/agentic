# TDD 驱动开发流程

> 本流程实现完整的测试驱动开发(TDD)范式：先写失败测试，再写最小实现，最后重构。
> 触发方式: `/tdd`
>
> **适用场景**: 新功能（纯逻辑模块、API、算法）、Bug 修复（自动诊断根因）。
> **不适用**: UI 原型探索、第三方 API 集成调试、一次性脚本。
>
> **智能模式**: 修 Bug 时，本流程会自动判断根因是否已知。未知时先走中立诊断，再无缝进入 TDD 循环。

## 步骤

1. **确认 TDD 基础设施**
   - 确认项目使用的测试框架（如 pytest、jest、vitest、go test 等）
   - 确认测试运行命令和配置文件
   - 确认测试文件目录结构和命名约定
   - 如无现有测试基础设施，先协助搭建最小化测试环境

2. **智能诊断（Bug 修复时自动触发）**
   如果用户描述的是一个 Bug 或异常行为，先进行根因评估：

   **判断标准 — 根因是否已知？**
   - ✅ **已知**: 用户或错误信息已明确指出具体代码位置和原因 → 跳到步骤 3
   - ❌ **未知**: 只有症状（报错、异常行为），不知道为什么 → 执行以下诊断

   **中立诊断流程**（遵循 `/debug` 的中立提示原则）：
   - 使用**中立提示词**追踪执行流（"跟随代码逻辑从入口到出错点，逐步汇报观察结果"）
   - **禁止**使用预设偏见提示（"帮我找 bug"、"这段代码有什么问题"）
   - 📊 如果代码图谱可用，辅助定位问题范围：
     - `query_graph(pattern="callers_of", target="<出问题的函数>")` — 查上游调用方
     - `query_graph(pattern="callees_of", target="<出问题的函数>")` — 查下游依赖
     - `query_graph(pattern="tests_for", target="<出问题的函数>")` — 查已有测试覆盖
   - 收集客观证据：错误日志、堆栈追踪、数据流、状态变化
   - 得出根因后，向用户简要确认诊断结论，再进入下一步

   > 如果任务是**新功能开发**（非 Bug 修复），直接跳到步骤 3。

3. **定义测试清单 (Test List)**
   - 📊 如果代码图谱可用，先查询已有测试覆盖：
     - `query_graph(pattern="tests_for", target="<目标函数/类>")` — 避免重复编写已有测试
     - 将已有测试标记为“已覆盖”，只编写缺失的测试用例
   - 基于需求（或诊断出的根因），分解为具体的行为描述
   - 将每个行为转化为一个测试用例描述（不写实现）
   - 按以下优先级排序：
     1. 核心正常路径（最简单的成功场景）/ Bug 复现测试
     2. 边界条件（空值、零值、上下限）
     3. 异常路径（非法输入、错误状态）
     4. 交互场景（模块间协作）
   - 测试清单提交用户确认

4. **逐项执行 TDD 循环**
   对测试清单中的每一项，严格按以下三步执行：

   🔴 **Red — 编写失败测试（必须先于任何实现代码）**
   - > ⛔ **铁律：无失败测试 → 禁止写实现代码。** 在没有对应的失败测试之前，任何实现文件的编辑都是违规的。
   - 编写一个**最小的**、**精确描述预期行为**的测试
   - 运行测试，**确认测试失败**（且失败原因正确：不是语法错误，而是功能未实现）
   - 如果测试意外通过 → 说明需求已被覆盖或测试写错了，需要审视

   🟢 **Green — 编写最小实现（仅满足当前测试，不多写一行）**
   - 编写**刚好让测试通过**的最少代码（可以是硬编码、丑陋但正确的代码）
   - 运行测试，**确认测试通过**
   - 同时运行已有的全部测试，确认无回归
   - **过度实现检查**：以下情况均为违规，必须删除多余代码：
     - ✖️ 测试没要求但“预期会用到”的函数/方法
     - ✖️ 提前写好的工具函数、常量定义、类型声明（除非当前测试需要）
     - ✖️ “顺便”添加的输入验证、日志、错误处理（当前测试未覆盖的）
   - **判断标准**: 删除这段代码后测试仍然通过？如果是，它就不应该存在

   🔵 **Refactor — 安全重构**
   - 在全量测试绿灯保护下，改善代码质量：
     - 消除重复、提取函数/方法、改善命名、简化逻辑
   - 📊 如果代码图谱可用，重构前检查影响范围：
     - `get_impact_radius(changed_files=[<重构涉及的文件>])` — 确认重构不影响外部调用方
     - 如果影响半径超出预期，谨慎重构或向用户确认
   - 运行全量测试，**确认仍然全部通过**
   - **铁律**: 重构不改变外部行为，只改善内部结构
   - 如果此步可跳过（代码已足够清晰），直接进入下一个测试

   ↩️ **循环**: 回到 🔴，取测试清单下一项，重复直到清单全部完成

5. **集成验证**
   - 运行全量测试套件，确认 0 failures
   - 检查测试覆盖范围是否满足需求
   - 如有遗漏的边界条件，补充测试并走 🔴→🟢→🔵 循环

6. **汇报 TDD 结果**
   向用户汇报：
   ```
   TDD 结果:
   - 诊断: [已知根因 / 经诊断发现: ...]（仅 Bug 修复时显示）
   - 测试清单: X 项全部完成
   - 测试通过: X/X 通过, 0 失败
   - TDD 循环数: X 轮 Red-Green-Refactor
   - 测试命令: [实际运行的命令]
   - 覆盖范围: [已覆盖的行为列表]
   - 代码质量: [重构摘要，如提取了哪些函数、消除了哪些重复]
   ```
