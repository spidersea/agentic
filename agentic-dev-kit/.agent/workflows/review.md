# 对抗式代码审查流程

> 本流程使用 SKILL.md 中的对抗验收模式进行深度代码审查。
> 触发方式: `/review`

## 步骤

1. **确认审查范围**
   - 向用户确认需要审查的文件或模块
   - 只加载相关文件（≤5 个），避免上下文膨胀
   - 加载审查标准 `.agent/rules/code-review.md`（5 维度审查清单）

2. **中立阅读**
   - 先以中立方式通读代码：
     - "阅读这些文件的输入、处理、输出，描述它们实际做了什么"
   - 记录代码的实际行为（不做任何质量判断）

3. **专家 A — 风险扫描（极度敏锐）**
   切换为敏锐的风险扫描视角：
   - 尽可能找出所有可能的风险点和代码瑕疵
   - 包括：安全漏洞、性能问题、边界条件、错误处理缺失、可维护性问题
   - 允许误报 — 目标是产出一份**风险超集**
   - 给每个风险点标注严重等级（低/中/高/关键）

4. **辩手 B — 严格反驳（极度严格）**
   切换为严格的反驳视角：
   - 逐条尝试推翻专家 A 的风险点
   - 为每个风险点寻找合理解释或证明其为误报
   - 引用具体代码逻辑、上下游保护、框架特性等作为证据
   - 目标是产出一份**真实缺陷子集**

5. **裁判 C — 最终裁定（掌握真相）**
   切换为客观裁判视角：
   - 根据代码的实际逻辑和运行行为，裁定每一条争议
   - 只有具备客观证据的问题才被标记为"已确认"
   - 产出**最终审查报告**：
     ```
     ## 已确认问题
     - [问题描述] — 严重等级: [等级] — 建议修复: [方案]
     
     ## 已排除（误报）
     - [风险点] — 排除理由: [证据]
     
     ## 需要人类决策
     - [争议点] — 原因: [为什么无法自动裁定]
     ```

6. **提交审查结果**
   - 将最终审查报告提交给用户
   - 对已确认问题，提出具体的修复建议
   - 对需要人类决策的争议点，提出 2-3 个选项供用户选择

7. **联动修复（如有已确认问题）**
   如果审查发现了已确认问题且用户同意修复：
   - 执行 `/debug` 流程修复问题（含举一反三 + 回测）
   - 修复完成后执行 `/test` 验证全量测试通过
