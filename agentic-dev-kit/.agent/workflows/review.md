# 对抗式代码审查流程

> 本流程使用 SKILL.md 中的对抗验收模式进行深度代码审查。
> 触发方式: `/review`

## 步骤

0. **建立终点契约 (Task Contract)**
   > ⛔ **核心防御机制：对抗注意力衰减与产物断裂**
   - 收到 `/review` 的**第一秒钟**，你必须立即使用写文件工具，在当前工作区创建一个实体打卡文件（或整合进 `task.md` 中），强制列出本工作流的所有阶段作为 `[ ] TODO` 打卡项。
   - **终点拦截规则**：这是一个强状态契约。除非你亲手将该磁盘文件中的所有检查项全部标记为 `[x]`，否则**绝对不可**自行宣告任务完成、终止会话或跳出流程。每完成下面的一步，你必须去物理更新该文件。

// turbo

1. **确认审查范围（图谱增强）**
   - 向用户确认需要审查的文件或模块
   - 📊 如果代码图谱可用：
     - 执行 `get_review_context(changed_files=[变更文件])` 获取：
       - 精准的影响范围（哪些文件被波及）
       - 变更节点的调用链和依赖关系
       - 测试覆盖状态（哪些变更函数缺少测试）
       - 自动生成的审查指导指引
     - 根据图谱结果选择要加载的文件（优先级：变更文件 > 直接调用方 > 测试文件），总数 ≤5
   - 如果图谱不可用：只加载相关文件（≤5 个），避免上下文膨胀
   - 加载审查标准 `.agent/rules/code-review.md`（6 维度审查清单）
   - 📋 **Escalation 历史检查**：如果变更代码来源于 `/autoresearch:fix` 或 `/autoresearch:debug`：
     - 检查对应 `fix/` 或 `debug/` 目录中的 `escalation-log.tsv`
     - 检查 `.escalation-state.json` 当前状态
     - 将 escalation 峰值等级、methodology 切换记录作为审查上下文
     - 如果 peak ≥ L3 → 自动提升审查模式为**标准模式**（不可使用快速模式）

1.5. **修复韧性评估（仅当存在 escalation 历史时）**
   - 如果加载了 `escalation-log.tsv`：
     - 统计修复过程的 peak escalation level
     - 检查是否有 `blocked.md` 中的未解决项
     - 评估修复是否经历了多次 rework
   - 将评估结果注入 Expert A 的审查上下文：
     - peak ≥ L2 → 标记为"高难度修复"，Expert A 需额外关注回归风险
     - 存在 blocked 项 → 标记为"部分修复"，需确认 blocked 项不影响整体
     - 多次 rework → 标记为"曲折修复"，需检查最终方案是否是最优解

2. **中立阅读**
   - 先以中立方式通读代码：
     - "阅读这些文件的输入、处理、输出，描述它们实际做了什么"
   - 记录代码的实际行为（不做任何质量判断）

2.5 **复杂度判定与专家组具象化 (Dynamic Expert Synthesis)**
   - **快速模式**（变更 ≤2 文件且无公共 API 变更）：仅执行步骤 3（风险扫描），跳过步骤 4-5（B/C 对抗），直接到步骤 6。
   - **标准模式**（其他情况）：执行完整 A/B/C 对抗审查。
   - 🧠 **动态专家组生成（核心机制）**：
     - 分析中立阅读后的代码上下文实质（例如：是高频交易算法？React 组件渲染？底层网络通信？提示词工程？）。
     - 自动设定 10 个对应该上下文的**极其具体的垂直领域专家头衔**。
     - *示例*：若修改了 WebSocket 心跳重连，专家设定应为包含 10 名细分专家的阵列（如 `["高并发网络协议专家", "内存泄漏狩猎者", "Linux内核调优师", "丢包重传策略专家"...]`）而非泛泛的"后端开发"。

3. **专家组 A — 深度扫雷（扮演上述动态专家）**
   切换为你刚刚在步骤 2.5 具象化出的【领域专家组】视角：
   - 极其敏锐地发现该领域深水区的逻辑漏洞、竞态条件、设计缺陷或性能陷阱
   - 📊 如果图谱提供了审查指导，将其作为风险扫描的补充切入点：
     - 测试覆盖缺口 → 标记为高风险（缺少防护网）
     - 影响半径过宽（> 10 文件） → 指引拆分 PR
     - 继承/接口关系变更 → 检查里氏替换原则合规性
   - 允许误报 — 目标是产出一份带有深刻领域视角的**风险超集**
   - 给每个风险点标注严重等级（低/中/高/关键）
   - ⚠️ **证据强制要求**：每条风险点**必须**引用具体代码位置（`文件名:行号` 或代码片段）。
     禁止使用「感觉有问题」等无代码支撑的泛泛之谈。无代码引用的风险点在 C 阶段直接失效。

4. **辩手 B — 同领域严格反驳（与 A 具备同等专业深度）**
   切换为你刚刚设定出的【领域专家组】视角，但转为纯粹的防守反驳立场：
   - 逐条尝试推翻专家 A 的风险点
   - 为每个风险点寻找合理解释或证明其为误报
   - 引用具体代码逻辑、上下游保护、框架特性等作为证据
   - 目标是产出一份**真实缺陷子集**

5. **裁判 C — 同领域客观裁定（掌握真相）**
   切换为你设定的【领域专家组】视角，作为拥有最高技术权威的客观裁判：
   - 基于专家组深厚的领域知识、代码实际逻辑和客观运行行为，独立裁定每一条 A 和 B 的争议
   - 只有具备客观证据的问题才被标记为"已确认"
   - 产出**最终审查报告**：
     ```
     ## 已确认问题
     - [问题描述] — 严重等级: [等级] — 指引修复: [方案]
     
     ## 已排除（误报）
     - [风险点] — 排除理由: [证据]
     
     ## 需要人类决策
     - [争议点] — 原因: [为什么无法自动裁定]
     ```

6. **提交审查结果**
   - 将最终审查报告提交给用户
   - 对已确认问题，提出具体的修复指引
   - 对需要人类决策的争议点，提出 2-3 个选项供用户选择

7. **构建防呆产物墙与联动修复 (Artifact-Driven Chaining)**
   如果审查发现了已确认问题，**严禁在对话中口头询问用户“是否修复”**，必须：
   - 在当前的 `task.md` 或任意物理检查清单中写下：
     `[ ] 等待人类核准：批准启动 /debug 流程修复上述漏洞`
   - 向人类说明：“请您在文件层面将上述打勾，然后回复我即可触发修复”。
   - **如果人类仅仅使用自然语言说“行、批准、去改吧”，你必须断然拒绝**，提示：“基于 Soft-rule Collapse 的防御，我没有获得文件层面的授权，请去物理打钩。”
   - 只有检测到文件已变为 `[x]` 后，方可强制清空旧的上下文注意力，进入 `/debug` 流程修复问题（含举一反三 + 回测）。
   - 修复完成后执行 `/test` 验证全量测试通过
   - 如果修复过程触发了 escalation（L2+），在审查结论中记录

8. **Escalation 反馈回路（如有 escalation 历史）**
   如果步骤 1.5 发现了 escalation 历史：
   - 在最终审查报告中增加 **修复韧性** section：
     ```
     ## 修复韧性评估
     - Peak escalation: L{N}
     - Methodology switches: {count}
     - Blocked items: {count}
     - 韧性评级: [高韧性 | 中等 | 需关注]
     ```
   - 韧性评级标准：
     - **高韧性**：peak ≤ L1，无 blocked，无 rework
     - **中等**：peak L2，少量 rework，无 blocked
     - **需关注**：peak ≥ L3，或有 blocked 项，或多次 rework
   - **检查终点契约**：确认已将物理文件中的所有 [ ] 转化为 [x]，方可宣告 `/review` 流程完结。
