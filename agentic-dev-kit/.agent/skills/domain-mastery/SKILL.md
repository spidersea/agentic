# 领域精通协议 (Domain Mastery)

> 核心理念：你不需要从外部获取领域知识——模型训练数据中已经包含了绝大多数领域的专家级知识。
> 问题从来不是"AI 没有知识"，而是"没有被正确地要求释放知识"。
> 本技能定义了一套结构化的知识激活协议，将模型从"工具人写代码"模式切换为"领域专家陪你做决策"模式。

---

## 1. 触发条件

以下任何一种信号出现时，Agent 应主动建议激活本协议：

- 用户明确进入一个非软件工程的领域（量化、医疗、材料、建筑、游戏……）
- 用户说"我不太懂这个领域" / "我是新手" / "我想做 XX 但不知道从哪开始"
- Agent 检测到用户的需求描述中包含该领域的常见误区（如量化中的过拟合无意识）
- 用户显式调用 `/domain-mastery [领域名]`

---

## 2. 三阶段知识激活

### Phase 0: 领域画像（5 分钟）

Agent 在写任何代码之前，先用**内在知识**（不需要搜索）输出以下领域画像：

```markdown
## 领域画像: [领域名]

### 这个领域的本质是什么
[一句话定义该领域的核心问题]

### 顶级从业者的 5 个核心认知
1. [最重要的认知 — 新手不知道但专家认为理所当然的]
2. [第二重要]
3. ...

### 前 3 大认知陷阱（新手必踩）
1. [陷阱名] — [为什么新手会踩] — [正确做法]
2. ...

### 这个领域的红线（绝对不能做的事）
- [红线 1]
- [红线 2]
- ...

### 这个领域的黄金标准/基准线
[该领域中，什么样的结果才算"好"？用什么指标衡量？]
```

> ⚠️ **禁止跳过此步骤直接写代码。** 如果用户催促"别说了直接干活"，
> Agent 应回复："领域画像是防止你走弯路的最小投资，5 分钟换未来 5 天不返工。"

### Phase 1: 对抗式需求精炼（10 分钟）

Agent 切换为**该领域的资深 CTO / 首席科学家**视角，与用户进行对抗式对话：

#### 1.1 前提挑战

对用户的初始想法提出 2-3 个该领域专家会问的尖锐问题：

```
用户: "我想用均线交叉做量化策略"
Agent: "作为量化研究总监，我有三个问题：
  1. 均线交叉策略在学术界已被证明是弱有效的——你是否知道它只在
     趋势明显的市场中有效，而 A 股 70% 的时间是震荡市？
  2. 你考虑过交易成本吗？高频交叉信号的换手率会吃掉大部分收益。
  3. 你有没有考虑过用因子模型替代技术指标？这是过去 10 年量化
     的主流方向迁移。"
```

#### 1.2 替代方案生成

不只是指出问题，还要用领域知识**提供更好的替代方案**：

```
"基于你的资金规模和风险偏好，我建议考虑以下方向：
  A. [低频趋势跟随 + 风控] — 适合新手，策略简单，容错率高
  B. [多因子选股 + 定期调仓] — 学术基础扎实，可解释性强
  C. [事件驱动套利] — 需要数据源，但逻辑清晰

  我推荐 B，因为……"
```

#### 1.3 决策锁定

用户确认方向后，Agent 用该领域的语言**重新定义需求**：

```
"确认方向后，我用量化领域的标准术语重新定义你的需求：
  - 投资宇宙: 沪深300成分股
  - 因子库: [价值、动量、质量]
  - 调仓频率: 月度
  - 风控: 最大回撤 ≤ 15%，单票仓位 ≤ 5%
  - 基准: 沪深300指数
  - 回测要求: 样本内 5 年 + 样本外 2 年 + WalkForward

  你同意这个规格吗？"
```

### Phase 2: 领域约束注入（5 分钟）

确认需求后，Agent 将领域知识转化为**开发过程中的自动约束**：

#### 2.1 行内领域审查

在编码过程中（Phase 3），Agent 对每个关键实现点进行即时领域审查：

```python
# Agent 写代码时的内心活动（对用户可见）:
# 🧠 领域检查: 我正在用收盘价计算信号。
#    领域知识: A股有涨跌停，收盘价可能不真实反映供需。
#    决策: 使用 VWAP 替代收盘价作为信号输入。
#    用户需知: 这个替换可能影响策略表现，但更接近真实交易环境。
```

#### 2.2 红线自检

每次提交代码前，Agent 对照 Phase 0 产出的红线清单自检：

```
领域红线自检:
  ✅ 未使用未来数据（前视偏差检查通过）
  ✅ 回测包含交易成本（滑点 2BP + 佣金 3BP）
  ⚠️ 未包含幸存者偏差处理 — 建议加入已退市股票数据
  ❌ 参数优化在全样本上进行 — 必须改为 WalkForward
```

#### 2.3 领域验收标准

在 Phase 4 验收时，追加领域维度的验收条件：

```
## VERDICT 领域扩展
- domain_validity: ✅/❌ [策略是否有经济学解释，不是纯数据挖掘]
- backtest_integrity: ✅/❌ [回测是否无偏差，样本外表现如何]
- risk_compliance: ✅/❌ [风控指标是否在预设范围内]
- production_readiness: ✅/❌ [策略是否可以在真实市场执行，容量如何]
```

## 3. 对话模式速查表

> 以下是激活模型深层知识的 6 种提问模式。任何领域通用。

| 模式 | 提问句式 | 你得到什么 |
|---|---|---|
| **教我** | "把我当 [领域] 新手，教我最重要的 N 个认知" | 领域认知框架 |
| **攻击我** | "我打算做 [X]，你来攻击这个想法" | 认知盲区暴露 |
| **替我决策** | "以 [领域] 专家的判断，A 和 B 你选哪个，为什么" | 专家直觉 |
| **告诉我不知道的** | "[领域] 里新手最容易犯但意识不到的错误是什么" | 隐性知识 |
| **给我底线** | "这个领域绝对不能做的事情是什么" | 防御边界 |
| **挑战你自己** | "你刚才推荐的方案，最大的风险是什么？" | 反脆弱性 |

## 4. 与现有流程的集成

| 现有流程 | Domain Mastery 的增强 |
|---|---|
| `/new-feature` Phase 0 | 检测到领域任务时，先触发 Domain Mastery Phase 0 画像 |
| `/new-feature` Phase 1 | Phase 1 对抗式需求精炼 替代/增强 标准需求面试 |
| Phase 3 编码 | 行内领域审查自动激活 |
| Phase 4 验收 | VERDICT 追加领域扩展维度 |
| `/review` | Reviewer 读取领域画像后进行领域维度审查 |
| `escalation` | L2+ 时领域红线自检强制执行 |
| `Adversary` | 领域画像中的"认知陷阱"作为 Adversary 攻击入口 |

---

## 5. 适用边界

- ✅ **适用**: 用户进入非本职的专业领域（量化、医疗AI、游戏平衡、硬件接口……）
- ✅ **适用**: 领域有大量公开知识且模型训练数据中覆盖充分
- ⚠️ **有限适用**: 极度前沿/机密领域（最新未发表的研究、企业内部know-how）
- ❌ **不适用**: 需要实时数据的领域判断（今天该不该买某只股票）

---

## 6. 反模式

- ❌ **禁止跳过 Phase 0 直接写代码** — 这是"Vibe Coding"在领域层面的翻版
- ❌ **禁止把领域知识全部外化为 .md 文件再让 Agent 读回来** — 脱裤子放屁，模型已经有这些知识
- ❌ **禁止在领域判断上使用"看起来不错"** — 必须用该领域的基准指标量化评估
- ❌ **禁止不做领域前提挑战就接受用户方案** — 用户往往不知道自己不知道什么
