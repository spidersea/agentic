# 通用代码开发与执行技能

> 本技能定义了 AI 智能体在代码研发中**必须遵循**的全部规范。适用于所有项目的新功能开发、重构、Debug 和测试编写。
>
> 它致力于解决大模型三大核心病灶：**上下文过载**、**谄媚幻觉**、**边界模糊**，最大化智能体的编码与执行效能。
>
> **根本心法**: 不要追逐花哨的外部插件和工具链。真正有用的能力会被前沿公司内置到产品中。保持工具链精简，只用基础 CLI + 理解核心原则，就足以完成世界级的工作。

---

## 一、核心原则

1. **少即是多 (Minimal Context)** — 将"研究/发散"与"执行/编码"**严格分离**。每次执行只加载最直接相关的少数文件。
2. **精确指令 (Precise Implementation)** — 绝不接受"去搭个XX系统"这类含糊指令。所有执行指令必须精确到技术选型、库版本和核心逻辑流。
3. **消除谄媚 (Neutral Validation)** — 检查代码和 Debug 时使用**中立陈述**，避免引导 AI 制造不存在的问题。
4. **终点契约 (Task Contract)** — 除非所有验收条件全部通过，否则绝不可自行宣布完成。
5. **会话隔离 (Session Isolation)** — 每个独立任务在新会话中执行，防止跨任务上下文污染。
5.5. **状态跃迁令** — 执行 `/xxx` 意味着**硬状态重置**：清空冲动 → 重新查阅 workflow 原文 → 严格遵循起始约束。
5.6. **产物链完整性** — 🔴 完整模式下，Phase 1 的 `openspec/` 是 Phase 3 编码的**唯一合法依据**。
6. **规则卫生** — 定期清理合并 Rules/Skills/AGENT.md。规则膨胀导致性能退化。
7. **人类兜底** — 最终质量责任始终由人类承担。必须审查关键产出。

| 反模式 | 说明 |

## 三、标准操作流程 (SOP)

> **有界执行原则**: 每个 Phase 包含明确的进入条件、执行预算和退出标准。预算内未达标必须暂停上报。

### Phase 0: 复杂度路由（强制，所有任务入口）

| 信号 | 🟢 轻量 | 🟡 标准 | 🔴 完整 |
|---|---|---|---|
| 修改文件数 | ≤2 | 3-7 | 8+ |
| 公共 API 变更 | 否 | 可能 | 是 |
| 需求明确度 | 清楚 | 大方向清 | 模糊 |

| 阶段 | 🟢 轻量 | 🟡 标准 | 🔴 完整 |
|---|---|---|---|
| Phase 1 调研 | ⏭️ 跳过 | 📝 简化 | 📋 完整 |
| Phase 2 契约 | ⏭️ 跳过 | 📝 内联确认 | 📋 正式 CONTRACT |
| Phase 3 编码 | ✅ 直接执行 | ✅ 直接执行 | ✅ 直接执行 |
| Phase 4 验证 | ✅ 测试+自检 | ✅ 测试+自检 | ✅ 测试+A/B/C |
| 人类确认 | 0次 | 1次 | 2-3次 |

> 💡 能自己决定的不问用户。只有置信度 < 80% 或不可逆变更时才确认。

### Phase 1: 调研与技术定调

🟡 标准或 🔴 完整模式触发。产出精确的技术规格说明（选型+版本+逻辑流+文件清单+任务分解），持久化到文件系统。

⛔ **全局防断层契约**: 2+ 阶段任务开启前，以 `[ ] TODO` 物理写下所有 Phase 作为检查单。

### Phase 2: 确立验收契约

创建 `{TaskName}_CONTRACT.md`，必须含：自动化测试用例(正常+边界+异常) + 测试命令 + 预期结果。**禁止修改测试来通过验收。**

### Phase 3: 严格上下文编码

清理 → 按需加载 → 严格遵循规格 → 增量交付。

**硬规则**: ①完整性原则(Boil the Lake) — 不存在可接受的简化版 ②真实实现 — 功能代码禁用 mock/placeholder ③生产级 — 测试覆盖和错误处理不可以 deadline 为由跳过

### Phase 4: 中立验证与对抗验收

1. 自动化测试: 单元→集成→E2E，失败修代码不修测试
2. 中立审查: 用中立提示词，不带预设偏见
3. A/B/C 对抗: 专家A(风险超集) → 辩手B(推翻) → 裁判C(裁定)
4. 闭环序列: 测试闸门→契约核验→回归确认→变更摘要→人类审批

> 死锁保护: 某步持续失败超 3 次迭代，暂停上报人类。

## 五、Rules 与 Skills 的区分

| 类型 | 定义 | 用途 |
|---|---|---|
| **Rules** | 编码偏好与禁令 | 告诉 Agent **不要做什么** |
| **Skills** | 操作配方与流程 | 告诉 Agent **怎样做某事** |

> **提前预审**: 不确定 Agent 如何解决问题时，先让它写成 Skill，审查后再执行。
