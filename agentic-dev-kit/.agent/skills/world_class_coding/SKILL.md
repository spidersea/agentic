# 通用代码开发与执行技能

> 本技能定义了 AI 智能体在代码研发中**必须遵循**的全部规范。适用于所有项目的新功能开发、重构、Debug 和测试编写。
>
> 它致力于解决大模型三大核心病灶：**上下文过载**、**谄媚幻觉**、**边界模糊**，最大化智能体的编码与执行效能。
>
> **根本心法**: 不要追逐花哨的外部插件和工具链。真正有用的能力会被前沿公司内置到产品中。保持工具链精简，只用基础 CLI + 理解核心原则，就足以完成世界级的工作。

---

## 一、核心原则

1. **少即是多 (Minimal Context)**
   将"研究/发散"与"执行/编码"**严格分离**。切忌将不相关的历史上下文和备选方案带入编码执行阶段。每次执行只加载最直接相关的少数文件。

2. **精确指令 (Precise Implementation)**
   绝不接受"去搭个XX系统"这类含糊指令。所有执行指令必须精确到**技术选型、库版本和核心逻辑流**。如果连自己都说不清细节，先开一个研究会话去定型。

3. **消除谄媚 (Neutral Validation)**
   在检查代码和 Debug 时，使用**中立陈述**（如"遍历代码逻辑并汇报观察结果"），而不是预设前提（如"帮我找出 bug"）。避免引导 AI 制造不存在的问题。

4. **终点契约 (Task Contract)**
   任务有明确的终结边界。**除非所有验收条件全部通过，否则绝不可自行宣布完成。** 不达标不可关闭任务。

5. **会话隔离 (Session Isolation)**
   每个独立任务/契约应在新的会话中执行。禁止在一个长会话中串联多个不相关任务，防止跨任务上下文污染。

5.5 **状态跃迁令 (Hard Workflow Transition)**
   无论你是收到人类输入的 `/<指令>`，还是在现有流程中被要求联动执行另一个流程（如 `/review` 后转入了 `/debug`）。**绝对禁止**凭大脑记忆的残影直接开写代码！这里的“执行 `/xxx`”意味着一个**硬状态重置**。你必须：(1) 在心中清空当前积压的代码逻辑与“想要立刻秀出代码”的奖励冲动；(2) **必须使用工具重新查阅 `.agent/workflows/xxx.md` 的内容原文**；(3) 严格遵循其最新的起始约束（如创建物理契约等）。不查阅源工作流直接产生代码即视为幻觉与犯规。

5.6 **产物链完整性 (Artifact Chain Integrity)**
   在 🔴 完整模式或 🟡 标准模式（跨 5+ 文件）下，Phase 1 产出的 `openspec/` 或设计文件，是 Phase 3 编码的**唯一合法依据**。禁止以"写了 implementation_plan.md 就算完成了 Phase 1"来替代 `/spec:propose` 产物。系统级工具（如 implementation_plan.md, task.md, walkthrough.md）是**执行追踪产物**，不替代**规范设计产物**（openspec/）。两者必须同时存在。如果你跳过了 `/spec:propose`，你必须在进入 Phase 3 前向用户承认遗漏并补充执行。不允许用"时间紧迫"或"规格已在脑中"来合理化跳过。

6. **规则卫生 (Rule Hygiene)**
   定期清理和合并 Rules、Skills 及 AGENT.md 中的条目，移除过时或互相矛盾的规则。规则膨胀会导致智能体性能退化。当规则超过 20 条或 Agent 需要读取超过 5 个 markdown 文件才能开始工作，说明已经到了必须清理的时刻。

7. **人类兜底 (Own The Outcome)**
   没有任何 Agent 是完美的。可以将大量设计和实现工作委托给 Agent，但最终的质量责任始终由人类承担。必须审查关键产出，不可盲信。

## 二、任务提示速查模板 (Prompt Quick Reference)

> 在向 Agent 下达任务前，使用此模板组织你的提示。覆盖越完整，Agent 产出越可靠。

```
- 目标 (Goal):       你想要改变什么或建立什么？
- 上下文 (Context):   哪些文件、文档、错误信息与此任务相关？（≤5 个）
- 约束 (Constraints): 需遵循的标准、架构、技术选型或惯例
- 完成条件 (Done-when): 任务完成前应满足什么条件？（测试通过、行为变化等）
```

**简易判断**: 如果你的 prompt 缺少上面任意一项，停下来补齐再发送。

## 三、反模式清单

> 以下做法将严重降低智能体效能，**必须坚决避免**。

- ❌ 一次性给 Agent 加载超过 5 个与当前任务无关的文件
- ❌ 使用预设偏见的 Debug 提示词（如"帮我找出这个 bug"、"这段代码有什么问题"）
- ❌ 在一个会话中串联超过 3 个不相关任务（上下文污染）
- ❌ 在 AGENT.md 中堆积上百条未分类的规则（规则膨胀）
- ❌ 给 Agent 需求描述时只说"搭个XX系统"而不给技术细节（模糊指令）
- ❌ 上下文被压缩/重置后，不重新读取任务计划就继续编码（记忆残影）
- ❌ Agent 自行引入未经授权的第三方依赖或"脑补"未定义的需求
- ❌ 未经检查点确认就在长任务中连续编码超过 3 个阶段（检查点缺失）
- ❌ 修 bug 或加功能时顺手改了不相关的代码格式、注释、命名或逻辑（**顺手重构 Drive-by Refactoring**）。**判断标准**: 每一行变更都必须可直接追溯到用户请求。发现不相关问题时口头报告，不动手删改。
- ❌ 用自己偏好的代码风格覆盖现有项目风格，如强行加类型注解、改引号风格、重排 import（**风格漂移 Style Drift**）。**解法**: 匹配现有代码风格，即使你觉得自己的更好。
- ❌ 删除与当前任务无关的预存死代码或注释（**擅删死代码 Unauthorized Dead Code Removal**）。**解法**: 只清理你本次改动导致的孤儿代码（未使用的 import/变量/函数）。预存死代码报告即可，不可擅自删除。

### 经典反模式命名

> 来自 CLAUDE.md 体系的经典命名，便于快速识别和沟通。

- 🍳 **厨房水槽会话 (Kitchen Sink Session)** — 一个会话中混杂多个不相关任务，上下文充满无关信息。**解法**: 不相关任务之间使用 `/context-reset` 清理上下文。
- 🔄 **反复纠正螺旋 (Correction Spiral)** — Agent 犯错后反复纠正仍未解决，上下文被失败方案污染。**解法**: 连续纠正 2 次仍失败时，执行 `/context-reset`，用更好的初始 prompt 重新开始。
- 📖 **过度膨胀的规则手册 (Bloated Rulebook)** — AGENT.md 或规则文件太长，Agent 开始选择性忽略指令。**解法**: 使用 `/evolve` 进行 Spa Day 清理，无情精简，删除 Agent 已自动遵守的规则或转为 `.agent/rules/` 模块化文件。

## 四、标准操作流程 (SOP)

> **有界执行原则 (Bounded Execution)**: 每个 Phase 都是一个有界循环，包含明确的进入条件、执行预算（步骤上限）和退出标准。如果在预算内未达标，必须暂停并上报人类决策，而非无限循环。

### Phase 0: 复杂度评估与自主度路由（强制，所有任务入口）

**任何任务开始前，Agent 必须先在内部（不问用户）评估复杂度并选择执行档位：**

| 信号 | 🟢 轻量 | 🟡 标准 | 🔴 完整 |
|---|---|---|---|
| 预计修改文件数 | ≤2 | 3-7 | 8+ |
| 涉及公共 API 变更 | 否 | 可能 | 是 |
| 需求明确度 | 用户已说清具体改什么 | 大方向清楚，细节需确认 | 模糊/空泛 |
| 架构影响 | 无 | 局部 | 跨模块/全局 |
| 典型场景 | 修 bug、改配置、加字段、调样式 | 新 API、重构模块、加功能页 | 新系统、架构变更、技术迁移 |

**三档行为差异：**

| 阶段 | 🟢 轻量 | 🟡 标准 | 🔴 完整 |
|---|---|---|---|
| Phase 1 调研 | ⏭️ **跳过** — 直接做 | 📝 简化：方案 + 1 个关键确认 | 📋 完整四阶段 |
| Phase 2 契约 | ⏭️ **跳过** — 改完跑测试即可 | 📝 简化：验收标准内联确认 | 📋 正式 CONTRACT.md |
| Phase 3 编码 | ✅ 直接执行 + 增量验证 | ✅ 直接执行 + 增量验证 | ✅ 直接执行 + 增量验证 |
| Phase 4 验证 | ✅ 跑测试 + 中立自检 | ✅ 跑测试 + 中立自检 | ✅ 跑测试 + A/B/C 对抗 |
| 人类确认次数 | **0**（做完汇报） | **1**（方案确认） | **2-3**（设计+契约+闭环） |
| `/spec:propose` | ❌ 不执行 | ⚠️ 跨 5+ 文件时可选 | ✅ 强制 |

**自主决策原则：**
- 🟢 轻量：Agent **自行做所有决策**，做完一次性汇报结果
- 🟡 标准：Agent **只在关键分歧点确认**（"推荐 A 因为 X，确认？"），其余自决
- 🔴 完整：按完整四阶段 SOP 执行
- ⚠️ 执行中发现比预期复杂时，**自动升档**并告知用户

> 💡 **核心理念**：能自己决定的不问用户。只有当 Agent 真正不确定（置信度 < 80%）或涉及不可逆变更时，才需要确认。

### Phase 1: 调研与技术定调 (Research & Scoping)

**触发条件**: 🟡 标准或 🔴 完整模式。🟢 轻量模式跳过此阶段。

> ⛔ **全局防断层契约 (SOP Enforcer)**
> 在开启任何包含 2 个阶段以上的编码任务之前，必须在你个人的 `task.md` 或当前工作区中以 `[ ] TODO` 形式物理写下所有即将执行的 Phase（如 调研、契约、编码、验证），作为强制跟踪检查单。
> 在此检查单全部被标记为 `[x]` 前，绝对不允许自行宣告完成、终止会话或被局部逻辑截断。

**动作**:
- **需求面试 (Requirements Interview)**:
  - 🟡 **标准模式**：Agent 自行研究，只在关键决策点请求 1 次确认（参见 `new-feature.md` 步骤 2 🟡 标准模式）
  - 🔴 **完整模式**：如果需求模糊或用户说不清细节，Agent 应**主动向用户提问**（而不是自行假设）。
  具体做法：提出 3-5 个关键问题，挑战用户假设，将模糊想法定型为具体方案。
  > 例：用户说“搭个认证系统” → Agent 应追问：JWT 还是 Session？是否需要 OAuth？密码策略？Token 过期时间？
- 开辟独立的**研究上下文区域**，不对具体代码进行修改，仅收集实现途径和对比优劣。
- 产出**精确的技术规格说明**，必须包含：
  - 技术选型与版本（如 "React 18 + TypeScript + Vite"）
  - 核心逻辑流设计
  - 模块依赖关系
  - 已知约束与边界
  - **文件清单**: 每个将创建/修改/测试的精确文件路径
  - **任务分解**: 拆分为 2-5 分钟的小任务，每个任务**必须包含**：
    - 精确文件路径（Create / Modify / Test）
    - **完整代码**（不可写"添加验证逻辑"等模糊描述，必须给出可直接粘贴的代码）
    - **精确运行命令 + 预期输出**（如 `pytest tests/auth/ -v` → 预期 PASS）
    - 验证步骤（如何确认这一步成功）
- 将技术规格保存为设计文件并 `git commit`（确保规格不仅存在于聊天记录中）

**退出标准**: 得到一份**极度精确、无歧义**的技术实施清单，已持久化到文件系统。

**⛳ 检查点 CP-1**: 技术规格说明已产出并确认 → 记录到状态文件。

### Phase 2: 确立验收契约 (Establish Task Contract)

**触发条件**: 准备进入实质性编码阶段前。

**动作**:
- 创建 `{TaskName}_CONTRACT.md` 或在实施计划中明确列出验收清单。
- 清单**必须**包含：
  - **自动化测试用例**（必须）:
    - 先定义测试用例（Test-First），再编写实现代码
    - 列出具体的测试命令（如 `npm test`、`pytest -v`）和预期结果
    - 测试必须包含：正常路径 + 至少一个边界条件 + 至少一个异常路径
  - **视觉断言**: 若为 UI，说明截屏需达到的精确表现形态
  - **行为断言**: 组件交互时的状态流转结果
  - **性能基线**: 如适用，定义可接受的性能指标
  - **测试运行命令**: 明确记录运行全量测试的确切命令

**强制规则**:
- 除非所有契约条件跑通（全绿测试 / UI 吻合），否则任务**绝不可标记完成**
- **禁止修改测试来通过验收** -- 除非测试本身有设计错误且经人类确认

**⛳ 检查点 CP-2**: 验收契约已创建并包含至少一个自动化测试用例 -> 记录到状态文件。

### Phase 3: 严格上下文编码 (Context-Strict Implementation)

**触发条件**: 技术规格与契约已拟妥，正式编写代码。

**动作**:
1. **清理多余记忆流**（Flush Context Bloat），重置大脑。
2. **按需加载**: 只允许读取本次开发最直接关联的少数文件。读取前务必再扫一眼"计划纲领"和"验收契约"。
3. **严格遵循**: 百分百遵循 Phase 1 确定的精确技术栈。不自行衍生未经授权的第三方依赖或"自作主张的脑补常识"。
4. **增量交付**: 每完成一个逻辑完整的模块，立即执行该模块相关的验证，不要等到全部编码完成后再统一验证。
5. **完整性原则（Boil the Lake）**: **必须始终选择完整的生产级实现**，不存在可接受的简化版/mock 版。
   - 测试覆盖和错误处理路径成本趋近于零，永远不可以 deadline 为由跳过
   - 如果功能过于复杂无法一次完成，应在 Phase 1 将其**拆分为多个独立可交付的子任务**，每个子任务本身必须是完整的生产级实现
   - 反模式：「先做简化版/mock 版，后续再完善」— **严格禁止**，这是 AI 时代之前的思维
6. **真实实现原则（No Mock in Production Code）**:
   - 功能代码必须使用真实依赖（真实的数据库连接、真实的 API 调用、真实的框架组件、真实的数据处理逻辑）
   - 禁止使用 mock 数据、硬编码假数据、placeholder 函数作为功能实现
   - 禁止 `# TODO: 替换为真实实现` 类注释 — 要么现在实现，要么在任务分解中明确标记为后续独立任务并上报用户
   - mock/stub **仅允许出现在测试文件中**，用于隔离外部依赖
   - 如果某个外部依赖暂时不可用，必须立即上报用户，而非用 mock 绕过

**⛳ 检查点 CP-3**: 每个模块编码完成后 → 运行该模块相关测试 → 记录通过/失败状态。

### Phase 4: 中立验证与对抗验收 (Verification & Adversarial Refereeing)

**触发条件**: 初版代码完成，开始执行契约验收。

**动作**:

#### 4.1 自动化测试验证
按以下顺序执行测试，每层通过后再进入下一层：

1. 运行单元测试 -- 验证函数/方法级别的正确性
2. 运行集成测试（如有） -- 验证模块间交互
3. 运行端到端测试（如有） -- 验证完整业务流程
4. 收集测试覆盖率（如适用） -- 与性能基线对比

**关键规则**:
- 如果测试失败，**必须修复代码而非修改测试**（除非测试设计有误）
- 修复后必须重新运行全量测试，确认未引入回归
- 测试结果必须以**实际命令输出**为准，不可凭记忆或推断

#### 4.2 中立代码审查
采用"中立型"提示词，不带预设偏见。

| ❌ 有偏见的提示 | ✅ 中立的提示 |
|---|---|
| "帮我找出这段代码的 bug" | "逐步跟随这段代码的执行逻辑，详述实际执行流并汇报观察结果" |
| "这个函数有什么问题" | "阅读这个函数的输入、处理、输出，描述它实际做了什么" |
| "为什么数据库查询这么慢" | "分析这段数据库查询的执行计划，汇报每一步的耗时和数据量" |
| "这段代码是否符合 JWT 规范" | "对照 JWT RFC 标准审查这段实现，汇报所有偏差和安全权衡" |

#### 4.3 深度 Debug — 对抗模式 (Adversarial Refereeing)
若遇到复杂漏洞或高风险逻辑，启用**左右互搏**：

- **专家 A（极度敏锐）**: 尽可能找出所有可能的风险点和代码瑕疵（包括误报）。目标是产出一份**风险超集**。
- **辩手 B（极度严格）**: 逐条尝试推翻专家 A 寻找的风险点，证明业务逻辑是可靠的。目标是产出一份**真实缺陷子集**。
- **裁判官 C（掌握真相）**: 根据客观的日志/运行结果/文件内容，裁定 A 和 B 的辩论，得出**最真实的 Bug 列表**。

#### 4.4 工作流闭环协议 (Workflow Closure Protocol)

修复真正被验证的缺陷后，执行以下**强制闭环序列**方可宣告任务完成：

**闭环序列（每一步都必须通过，否则不可标记完成）:**
1. **测试闸门** -- 运行全量测试套件，确认 0 failures
2. **契约核验** -- 逐条核对 CONTRACT.md 中的每个验收标准
3. **回归确认** -- 确认本次修改未破坏已有功能
4. **变更摘要** -- 生成本次变更的文件清单和关键改动说明
5. **人类审批门** -- 将闭环报告提交用户确认

**闭环报告模板**:
```markdown
## 工作流闭环报告
- **任务名称**: [TaskName]
- **测试结果**: [X/X 通过, 0 失败]
- **测试命令**: [实际运行的命令]
- **契约核验**:
  - [x] [契约条件1] -- 通过
  - [x] [契约条件2] -- 通过
- **变更文件**: [修改的文件列表]
- **回归状态**: [已有测试全部通过 / 有回归需关注]
- **风险提示**: [如有遗留风险或需要后续关注的事项]
```

**⛳ 检查点 CP-4**: 闭环序列全部通过 -> 记录闭环报告 -> 提交用户审批。

> 死锁保护: 如果闭环序列中某步持续失败超过 3 次迭代，必须暂停并上报人类，附上完整的失败日志和已尝试的修复方案。不可无限循环。

## 五、可持续节点协议 (Sustainability & Checkpoint Protocol)

> 此协议确保 Agent 在长任务、跨会话、上下文压缩等场景下**不丢失关键状态**，能可靠地断点续作。

### 4.1 状态分层模型

Agent 的状态分为三层，各层的持久化策略不同：

| 层级 | 名称 | 内容 | 持久化策略 |
|---|---|---|---|
| **L1** | 工作状态 (Working State) | 当前正在处理的任务、活跃的文件列表、当前 Phase | 每个检查点更新，写入状态文件 |
| **L2** | 事件记忆 (Event Memory) | 已完成的操作记录、测试结果、关键决策及理由 | 追加写入，不可修改历史 |
| **L3** | 身份配置 (Identity & Config) | 技术规格、验收契约、技能规范 | 缓慢变化，仅在 Phase 1 确定 |

### 4.2 检查点规范

**何时创建检查点：**
- ✅ 每个 Phase 完成时（CP-1 ~ CP-4）
- ✅ 每个逻辑模块编码完成且测试通过后
- ✅ 即将请求人类审批/输入前
- ✅ 执行代价较高的操作（大规模重构、数据库迁移）前
- ✅ 上下文即将被压缩/重置前

**检查点内容模板：**
```markdown
## Checkpoint: [CP-ID] — [Phase 名称]
- **时间**: [时间戳]
- **当前 Phase**: [Phase 1/2/3/4]
- **已完成**: [已完成的工作摘要]
- **待完成**: [剩余工作清单]
- **关键文件**: [当前正在修改的文件列表]
- **测试状态**: [通过/失败/未运行]
- **决策记录**: [本阶段做出的关键技术决策及理由]
- **下一步**: [明确的下一步行动]
```

### 4.3 上下文恢复协议 (Context Recovery)

每次上下文被压缩/重置/跨会话恢复时，**禁止凭"记忆残影"继续工作**。必须执行以下恢复序列：

```
恢复序列（严格按顺序执行）:
1. 读取 AGENT.md（获取路由和强制规则）
2. 读取本技能文件 SKILL.md（恢复行为规范）
3. 读取最新的检查点/状态文件（恢复 L1 工作状态）
4. 读取验收契约 {TaskName}_CONTRACT.md（恢复 L3 终点定义）
5. 读取当前正在修改的核心文件（恢复编码上下文）
6. 对比检查点中的"待完成"清单，确认当前位置
```

**确认上下文完整后，才可继续编码。**

### 4.4 交接协议 (Handoff Protocol)

当任务需要从一个会话传递到另一个会话时，产出一个**交接备忘录**，只包含后继会话最需要的信息：

```markdown
## 任务交接备忘录
- **任务名称**: [TaskName]
- **当前状态**: [Phase X, CP-Y 已通过]
- **核心背景** (≤5 句): [后继会话最需要知道的核心上下文]
- **关键文件**: [需要读取的文件列表，不超过 5 个]
- **已验证通过的决策**: [不需要重新讨论的技术决策]
- **待解决问题**: [具体的、可执行的下一步]
```

### 4.5 会话管理决策树 (Session Management)

> 在开始新工作或遇到上下文膨胀时，按以下决策树选择正确的会话策略：

| 场景 | 操作 |
|---|---|
| 工作仍属同一问题 | 保持当前会话，继续推进 |
| 工作出现真正的独立分支 | fork 新线程，保留原始上下文 |
| 完全无关的新任务 | 开新会话（会话隔离原则） |
| 上下文过长但仍需继续 | compact 压缩早期上下文后继续 |
| 有可并行的独立子任务 | 分派给子 Agent，主 Agent 专注核心问题 |
| 连续纠正 2 次仍失败 | 停止！执行 context-reset，用更好的初始 prompt 重新开始 |

**核心原则**: 一个线程 = 一个连贯的工作单元。当线程的关注点开始分散时，就是该拆分的时候。

## 六、防御性提示设计 (Defensive Prompting)

> 在与 Agent 交互时，使用以下模式来最大化输出质量、最小化幻觉。

### 5.1 角色锚定
在任务开始时明确 Agent 的角色和边界：
```
你是一个专注于 [领域] 的工程师。你的工作范围严格限制在 [具体范围]。
你不可以 [明确禁止的行为]。如果遇到超出范围的问题，立即汇报而不是猜测。
```

### 5.2 输出格式约束
要求 Agent 以结构化格式输出关键决策：
```
对于每一个你做出的技术决策，必须以以下格式记录：
- 决策: [做了什么选择]
- 理由: [为什么这样选择]
- 替代方案: [考虑过但放弃的选择]
- 风险: [这个决策的潜在风险]
```

### 5.3 不确定性声明
要求 Agent 在不确定时主动声明，而非猜测：
```
如果你对某个实现细节不确定（置信度 < 80%），必须：
1. 明确标注 "[不确定]" 标签
2. 列出你不确定的具体点
3. 提供 2-3 个可能的方案供人类选择
绝对不要在不确定时自行假设并继续执行。
```

## 七、Rules 与 Skills 的区分

| 类型 | 定义 | 用途 | 示例 |
|---|---|---|---|
| **Rules（规则）** | 编码偏好与禁令 | 告诉 Agent **不要做什么**或**偏好什么** | "禁止使用 `any` 类型"、"日志必须用结构化格式" |
| **Skills（技能）** | 操作配方与流程 | 告诉 Agent **怎样做某件事** | 本文档、部署流程、测试编写规范 |

> **提前预审技巧**: 如果不确定 Agent 会如何解决一个问题，让它先研究并**写成 Skill**。可以在 Agent 实际执行前审查和修正它的解题思路。

## 十一、任务生命周期检查清单模板

在启动复杂任务时，使用此清单维护状态并确保每个检查点和闭环门禁被执行：

```markdown
### Phase 1: 调研与技术定调
- [ ] 研究现有状态，阅读相关知识项和代码
- [ ] 确定范围，识别瓶颈和约束
- [ ] 输出精确的技术实施清单（含技术选型、版本、架构）
- [ ] CP-1: 技术规格确认，记录到状态文件

### Phase 2: 确立验收契约
- [ ] 定义自动化测试用例（正常路径 + 边界条件 + 异常路径）
- [ ] 明确测试运行命令和预期结果
- [ ] 定义行为断言（特定条件下的系统行为）
- [ ] 定义性能基线（如适用）
- [ ] CP-2: 契约创建完成，含至少一个自动化测试用例

### Phase 3: 严格上下文编码
- [ ] 清理上下文，严格聚焦于技术规格
- [ ] 仅加载 Phase 1 定义的相关文件
- [ ] 按规格精确编码，不做未授权衍生
- [ ] 编写自动化测试（与功能代码同步）
- [ ] 每完成一个模块立即运行该模块测试
- [ ] CP-3: 各模块测试通过，记录状态

### Phase 4: 中立验证与对抗验收
- [ ] 运行全量自动化测试（按金字塔顺序：单元 -> 集成 -> E2E）
- [ ] 中立审查代码逻辑（使用中立提示词）
- [ ] 对抗验证复杂逻辑（A/B/C 模式）
- [ ] 修复验证发现的真实缺陷，重新运行全量测试

### 闭环验收
- [ ] 测试闸门: 全量测试 0 failures
- [ ] 契约核验: 逐条确认契约每项通过
- [ ] 回归确认: 已有测试未被破坏
- [ ] 变更摘要: 生成变更文件清单和关键改动说明
- [ ] 人类审批: 闭环报告提交用户确认
- [ ] CP-4: 闭环完成，记录最终状态
```
