# 自动化测试与 TDD 纪律

> 测试是 Agent 证明自己工作正确的**唯一客观方式**。没有测试的代码 = 未经验证的代码。

## 一、自动化测试纪律

### 1.1 测试优先原则 (Test-First)
- 在 Phase 2 契约中先定义测试预期，再在 Phase 3 编写实现
- 测试代码和业务代码同等重要，不可将测试视为"可选附加"
- 每个新功能 / Bug 修复必须伴随至少一个自动化测试

### 1.2 测试金字塔
- **单元测试**（大量）: 必须。每个核心函数至少一个正常路径 + 一个异常路径
- **集成测试**（适量）: 推荐。模块间有交互时编写
- **E2E 测试**（少量）: 视项目而定。关键用户流程编写

### 1.3 测试编写规范
- 测试命名清晰：`test_<功能>_<场景>_<预期结果>`
- 每个测试只验证一个行为（Single Assertion Principle）
- 测试之间相互独立，不依赖执行顺序
- 使用 mock/stub 隔离外部依赖（数据库、API、文件系统）
- **禁止**在测试中使用 `sleep` 或硬编码延迟

### 1.4 测试执行规则
- 修改代码后**必须运行相关测试**再提交
- 全量测试通过是工作流闭环的**前置条件**
- 测试失败时修复代码，**不可修改测试来绕过**（除非测试设计有误且经人类确认）
- 测试结果以**实际命令输出**为准，不可凭记忆推断

## 二、TDD 编程范式 (Test-Driven Development Paradigm)

> TDD 不只是"先写测试"，它是一种**以测试驱动设计**的编程范式。当启用 TDD 模式（通过 `/tdd` 工作流触发）时，以下规范为**强制要求**。

### 2.1 核心循环: Red-Green-Refactor

TDD 的本质是一个**纪律严明的微循环**，每轮只前进一小步：

```
🔴 Red:    写一个最小的失败测试  → 运行 → 确认失败（且原因正确）
🟢 Green:  写刚好让测试通过的代码 → 运行 → 确认通过 + 无回归
🔵 Refactor: 在绿灯保护下改善代码  → 运行 → 确认仍通过
↩️ 循环:   取下一个测试，回到 🔴
```

**每一步的纪律要求**：

| 步骤 | 必须做 | 禁止做 |
|---|---|---|
| 🔴 Red | 先运行测试确认**确实失败**；失败原因是功能未实现，而非语法错误 | 不可跳过运行直接写实现；不可一次写多个测试 |
| 🟢 Green | 写**最少代码**使测试通过（允许硬编码和丑陋代码）；同步运行全量测试确认无回归 | 不可写"测试未要求但预期会用到"的代码（YAGNI） |
| 🔵 Refactor | 仅改善内部结构（消除重复、提取函数、命名优化）；保持全量测试绿灯 | 不可在重构中引入新功能或改变外部行为 |

### 2.2 适用场景

**✅ 强烈推荐使用 TDD 的场景**：
- 纯逻辑模块（计算引擎、业务规则、状态机）
- API 端点开发（输入 → 处理 → 输出，边界清晰）
- 数据处理管道（ETL、格式转换、验证逻辑）
- 算法实现（排序、搜索、优化、信号处理）
- Bug 修复（先用测试复现 Bug，再修复，确保不回归）

**⚠️ 需要变通的场景**（可用 TDD 精神但不必严格三步循环）：
- UI 原型探索（先快速搭建，稳定后补充视觉/交互测试）
- 第三方 API 集成（外部行为不可控，优先使用集成测试或契约测试）
- 探索性编码（技术可行性验证阶段，验证后再补测试）
- 配置和声明式代码（纯配置文件无需 TDD）

### 2.3 TDD 反模式

> 以下做法破坏 TDD 纪律，必须避免。

- ❌ **假 TDD**: 先写完实现再补测试，事后声称"做了 TDD"
- ❌ **测试积压**: 一次性列出所有测试用例并全部写完，然后才开始写实现（正确做法：每次只写一个测试，循环推进）
- ❌ **黄金锤子**: 对所有代码无差别使用 TDD（配置文件、UI 原型不适合严格 TDD）
- ❌ **脆弱测试**: 测试过度依赖实现细节（如断言内部方法调用顺序），导致重构时大量测试失败
- ❌ **重构越界**: 在 Refactor 步骤中引入新功能或改变 API 接口（重构 ≠ 重写）
- ❌ **跳过红灯**: 不运行测试就开始写实现，失去了"红灯确认测试有效"的保障

### 2.4 TDD 与四阶段 SOP 的整合

TDD 范式可以无缝嵌入现有四阶段 SOP：

| SOP Phase | TDD 集成方式 |
|---|---|
| **Phase 1** 调研与技术定调 | 在技术规格中标注"本模块采用 TDD 开发"，确认测试基础设施 |
| **Phase 2** 确立验收契约 | 契约中的测试清单 = TDD 的 Test List（驱动循环的燃料） |
| **Phase 3** 严格上下文编码 | 编码方式从"写代码→跑测试"变为"🔴→🟢→🔵 循环"；每完成一轮循环即为一个增量交付 |
| **Phase 4** 中立验证 | 审查 TDD 纪律遵循度：是否每步都有红灯记录？是否有过度实现？重构是否安全？ |
