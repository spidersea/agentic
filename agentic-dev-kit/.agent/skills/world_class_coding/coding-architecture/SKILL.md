# 架构纪律 (Architecture Discipline)

> 以下是与项目无关的通用架构原则。AI 编码时**必须遵守**，除非项目规则明确覆盖。

### 1.代码设计原则
- **单一职责**: 每个函数/类/模块只做一件事。如果描述中出现"和"，应该拆分
- **最小接口**: 模块对外暴露最少必要的接口，隐藏内部实现细节
- **增量修改优于大规模重写**: 修改现有代码时，先理解现有逻辑，再做最小化改动。不要为了"看起来更好"而重写正常运行的代码
- **依赖方向单一**: 高层模块不应依赖低层实现细节，通过抽象接口解耦

### 2.错误处理
- **不吞异常**: 捕获异常后必须有明确处理（日志、上报、恢复），禁止空 catch
- **错误信息必须有上下文**: 包含什么操作、什么输入、在哪里失败
- **快速失败**: 在函数入口验证参数，发现问题立即抛出，不要等到深层逻辑才报错
- **统一错误模式**: 同一项目中使用一致的错误处理方式（如统一的 Error 类、错误码体系）

### 3.安全基线
- **绝不硬编码密钥**: 任何密钥、token、密码必须通过环境变量或配置管理
- **输入验证**: 所有外部输入（API参数、用户输入、文件内容）必须验证后再使用
- **最小权限**: 代码只请求完成任务所需的最小权限
- **敏感数据不入日志**: 禁止在日志中输出密码、token、个人信息

### 4.代码可读性
- **命名即文档**: 变量、函数、类的命名必须清晰表达意图，无需注释补充
- **注释说明"为什么"**: 注释不解释代码做什么（代码本身应该表达），而是解释为什么这样做
- **函数保持精简**: 单个函数不超过合理行数（约 50 行），超出时拆分为子函数
- **避免魔法数字**: 所有常量必须命名，不可直接在代码中使用裸数字

### 5.变更安全
- **理解后再改**: 修改任何现有代码前，必须先阅读并理解其当前逻辑和上下游依赖
- **最小化影响面**: 修改应尽可能局部化，不无故波及无关模块
- **保持向后兼容**: 除非明确要求，API 和接口的变更应保持向后兼容

### 6.工具使用优先级
> 正确的工具选择是系统稳定性的关键来源。

- **专用工具优先于通用 shell**：执行有专用工具的操作时（如读文件、编辑文件、搜索），必须使用专用工具而非 `cat`/`sed`/`grep` 等 shell 替代
- **声明式优于命令式**：能用声明式工具完成的操作（如结构化文件编辑），不要用命令式 shell 脚本拼凑
- **并行优先于串行**：没有依赖关系的多个工具调用必须并行发起，不要顺序等待
- **最小权限工具**：能用只读工具完成的任务，不要使用读写工具（如探索代码时用搜索而非打开编辑器）
- **先读后写**：修改任何文件前，必须先用读取工具查看其当前内容，禁止盲写
- **Bash 的合理场景**：以下情况应使用 shell 而非专用工具：
  - 安装/更新依赖（`npm install`, `pip install`, `cargo build`）
  - 运行构建和测试命令（`npm run build`, `pytest`, `make`）
  - 执行项目自定义脚本（`bash scripts/deploy.sh`）
  - 环境诊断（`which`, `env`, `uname`, `df`）
