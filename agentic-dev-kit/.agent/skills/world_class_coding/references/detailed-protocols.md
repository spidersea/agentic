# 详细协议参考

## 可持续节点协议 (Sustainability & Checkpoint Protocol)

> 此协议确保 Agent 在长任务、跨会话、上下文压缩等场景下**不丢失关键状态**，能可靠地断点续作。

### 状态分层模型

| 层级 | 名称 | 内容 | 持久化策略 |
|---|---|---|---|
| **L1** | 工作状态 | 当前任务、活跃文件、当前 Phase | 每个检查点更新 |
| **L2** | 事件记忆 | 操作记录、测试结果、关键决策 | 追加写入，不可改历史 |
| **L3** | 身份配置 | 技术规格、验收契约、技能规范 | 仅在 Phase 1 确定 |

### 检查点规范

**何时创建：** Phase 完成时(CP-1~4) / 模块测试通过后 / 请求人类审批前 / 高代价操作前 / 上下文压缩前

**检查点模板：**
```markdown
## Checkpoint: [CP-ID] — [Phase 名称]
- **当前 Phase**: [Phase 1/2/3/4]
- **已完成**: [已完成工作摘要]
- **待完成**: [剩余工作清单]
- **关键文件**: [正在修改的文件列表]
- **测试状态**: [通过/失败/未运行]
- **决策记录**: [关键技术决策及理由]
- **下一步**: [明确的下一步行动]
```

### 上下文恢复协议

每次上下文压缩/重置/跨会话恢复时，**禁止凭残影继续**。必须：
1. 读取 AGENT.md（路由+规则）
2. 读取本 SKILL.md（行为规范）
3. 读取最新检查点/状态文件（L1）
4. 读取验收契约（L3）
5. 读取当前修改的核心文件
6. 对比"待完成"清单确认位置

### 交接协议

```markdown
## 任务交接备忘录
- **任务名称**: [TaskName]
- **当前状态**: [Phase X, CP-Y 已通过]
- **核心背景** (≤5 句): [核心上下文]
- **关键文件**: [≤5 个文件]
- **已验证通过的决策**: [技术决策]
- **待解决问题**: [可执行下一步]
```

### 会话管理决策树

| 场景 | 操作 |
|---|---|
| 同一问题 | 保持当前会话 |
| 独立分支 | fork 新线程 |
| 无关新任务 | 开新会话 |
| 上下文过长 | compact 压缩后继续 |
| 可并行子任务 | 分派子 Agent |
| 连续纠正 2 次失败 | context-reset + 更好 prompt |

## 防御性提示设计

### 角色锚定
```
你是一个专注于 [领域] 的工程师。工作范围严格限制在 [具体范围]。
不可以 [明确禁止的行为]。超出范围的问题立即汇报不猜测。
```

### 输出格式约束
```
对于每个技术决策必须记录：决策 / 理由 / 替代方案 / 风险
```

### 不确定性声明
```
置信度 < 80% 时：标注 [不确定] → 列出具体点 → 提供 2-3 方案供选择
```

## 任务生命周期检查清单模板

```markdown
### Phase 1: 调研与技术定调
- [ ] 研究现有状态，阅读相关知识项和代码
- [ ] 确定范围，识别瓶颈和约束
- [ ] 输出精确技术实施清单
- [ ] CP-1: 技术规格确认

### Phase 2: 确立验收契约
- [ ] 定义自动化测试用例（正常+边界+异常）
- [ ] 明确测试命令和预期结果
- [ ] CP-2: 契约创建完成

### Phase 3: 严格上下文编码
- [ ] 清理上下文，严格聚焦技术规格
- [ ] 按规格编码 + 同步测试
- [ ] CP-3: 各模块测试通过

### Phase 4: 中立验证与对抗验收
- [ ] 全量测试（单元→集成→E2E）
- [ ] 中立审查 + 对抗验证
- [ ] 修复缺陷后重跑全量测试

### 闭环验收
- [ ] 测试闸门: 0 failures
- [ ] 契约核验: 逐条确认
- [ ] 回归确认 + 变更摘要
- [ ] 人类审批
- [ ] CP-4: 闭环完成
```
