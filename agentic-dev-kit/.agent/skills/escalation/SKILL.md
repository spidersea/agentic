# 失败压力升级系统（Escalation）

> 灵感来源：[tanweai/pua](https://github.com/tanweai/pua) 的 L1-L4 压力升级机制。
> 核心理念：根据连续失败次数自动升级行为约束，迫使 Agent 穷尽一切方案后才允许放弃。

> ⛔ **压力状态锁存 (Escalation Locking)**
> 在进入任何一层 Escalation（尤其是 L3/L4）之前，必须物理锁定当前尝试解决的具体 Checklist 到当前工作区文件（如 `task.md`）。
> 在该 Checklist 中的所有项全部物理打满 `[x]` 前，不允许自行放弃，更不允许静默回退到正常模式。

## 设计原则

1. **渐进式压力**：失败越多，约束越严，行动要求越具体
2. **方法论驱动**：不是简单的重试，而是强制切换解决思路
3. **闭环验证**：每次操作必须有可验证的输出证据
4. **体面退出**：穷尽一切后允许结构化的失败报告

## 压力等级

| 连续失败次数 | 等级 | 标识 | 强制动作 | 邪修联动 |
|------------|------|------|---------|----------|
| 2 | **L1 切换方案** | `[L1 ⚡]` | 切换**本质不同**的方案（换参数/换函数名不算） | + 对抗人格分析（邪修一式） |
| 3 | **L2 深度调查** | `[L2 🔍]` | 搜索 + 读源码上下文 50 行 + 列 3 个假设 + 建议切换方法论 | + 推理接力 3 快照（邪修二式） |
| 4 | **L3 强制清单** | `[L3 📋]` | 完成下方**七项检查清单**（每项必须打勾） | + 委派 Adversary Agent 攻击假设（邪修四式） |
| 5+ | **L4 最后手段** | `[L4 🚨]` | 强制切换方法论 + 拼命模式（参见 `autoresearch/references/methodology-router.md`） | + Polanyi 挖掘 + 工具自造（邪修五+三式） |
| 7+ | **L5 Mythos 模拟** | `[L5 👁️]` | 全能力释放模式（见下方 L5 详细定义） | 全邪修式并行激活 |

### 等级判定规则

- **失败定义**：命令执行返回非零 exit code、测试失败、构建失败、功能验证不通过
- **连续计数**：成功执行一次后重置为 0
- **不计入失败**：首次尝试失败（允许一次试错）、已知修复正在执行中
- **强制升级**：如果同一思路连续微调 3 次（换参数/换库版本/换格式），视为"原地打转"，直接跳到 L2
- **知识驱动决策**（内容层联动）：
  - L2+ 时**必须**查阅 `.agent/state/memory-palace/failure-patterns.jsonl` 匹配历史失败模式，避免重蹈覆辙
  - L3+ 时**必须**查阅 `.agent/state/captured-patterns/` 匹配已知项目模式
  - 如果历史模式命中 → 直接应用历史 resolution，跳过盲猜阶段

## 七项检查清单（L3+ 强制完成）

达到 L3 时，**必须逐项确认**以下清单，跳过任何一项即禁止声称"无法解决"。

> ⚡ **隐性直觉豁免权 (Tacit Intuition Leap) [Polanyi 升级]**：
> 如果你产生了一个基于深层代码感觉的极强假设，并且能清晰表述其“为什么本质上突破了目前的框架”，你可以跳过官僚化的清单检查直接执行该跳跃。**代价是**：这是一个带有高度风险的认识论承诺（Hazardous Commitment）。如果跳跃后仍失败，**必须无条件直接坠入 L4 拼命模式**，丧失任何缓冲权。

- [ ] **逐字读完失败信号**：逐行阅读完整错误信息、stack trace、日志输出
- [ ] **用工具搜索过核心问题**：用报错原文 / 官方文档 / 多角度关键词搜索
- [ ] **读过失败位置的原始上下文**：读源码上下文 50 行（不是摘要或记忆）
- [ ] **所有假设都用工具确认**：版本号、路径、权限、依赖关系 — 用命令验证
- [ ] **试过完全相反的假设**：一直假设"问题在 A" → 现在假设"问题不在 A"
- [ ] **能在最小范围内复现问题**：构造最小可复现用例
- [ ] **换过工具/方法/角度/技术栈**：本质不同的方案（非微调）

## 抗合理化表

当 Agent 产生以下"合理借口"时，自动触发对应等级的约束：

| 借口模式 | 反击（内部自检提示） | 触发等级 |
|---------|-------------------|---------|
| "超出能力范围" | 七项清单完成了吗？方法论走完了吗？ | L1 |
| "建议用户手动处理" | 这是你的任务。Owner 意识 — 谁的问题谁解决 | L3 |
| "已尝试所有方法" | 列出尝试过的完整清单 — 少于 3 种不算穷尽 | L2 |
| "可能是环境问题" | 这如果是惰性借口，请用命令验证；若是真实的**认识论阻力**（环境根本性损坏/工具断裂），转入 Tooling Breakdown Reassessment 分析工具本身，而非死磕代码。 | L2 或对齐模式 |
| "需要更多上下文" | 你有工具。若是因为系统太复杂潜规则太多而迷失，请主动读取或构造 `tacit-tradition-map.md`。 | L2 |
| 反复微调同一处 | 换参数不叫换方案。三次同思路直接 L2 | L1→L2 |
| "我无法解决" | 七项清单全部完成了吗？（RedLine-3：未穷尽禁止放弃） | L4 |
| 空口说"已完成" | 证据呢？build 跑了吗？test 通过了吗？贴输出 | L2 |
| 等用户指示下一步 | 主动出击 — 分析可能的下一步，提出方案 | 能动性鞭策 |
| 修完不验证就跑 | 承诺的结果用证据交付。跟到底 | L1 |
| 修了 A 破坏了 B | 改之前跑过全量测试了吗？回归测试是底线 | L2 |

## 通用方法论（五步法）

当陷入卡壳时，**按序执行**以下步骤：

1. **闻味道** — 列出所有尝试过的方案，找共同模式。同一思路微调 = 原地打转
2. **揪头发** — 按序执行（禁止跳过）：
   - 逐字读失败信号
   - 主动搜索（报错原文 / 官方文档 / 多角度关键词）
   - 读原始材料（源码上下文 50 行，不是摘要）
   - 验证前置假设（版本、路径、权限、依赖 — 用工具确认）
   - 反转假设（一直假设"问题在 A" → 现在假设"问题不在 A"）
3. **照镜子** — 是否在重复？是否该搜索却没搜？是否忽略了最简单的可能？
4. **执行新方案** — 必须与之前**本质不同**，有明确验证标准
5. **复盘** — 解决后检查同类问题 + 修复完整性 + 预防措施

> ⚠️ 步骤 1-4 完成前必须不向用户提问 — 除非需求本身就是模糊的，先澄清再执行。

## L5 Mythos 模拟模式（连续 7+ 次失败）

当 L4 仍然无法解决时，进入全能力释放模式——模拟 Mythos 级别的对抗推理能力：

1. **全邪修式并行激活**：同时启动对抗人格 + 推理接力 + Polanyi 挖掘 + 工具自造
2. **Adversary Agent 权限升级**：从 ReadOnly 升级为 WorkspaceWrite，允许构造攻击性测试用例
3. **3 路并行推理**：三个不同假设同时推进，每路独立的 reasoning-relay 文件
4. **强制查阅外部知识**：web search + doc-lookup 多角度搜索
5. **记忆宫殿全量扫描**：读取 `.agent/state/memory-palace/` 中所有历史失败模式

> ⚠️ **L5 代价**：token 消耗激增 ~5x，仅用于真正的硬核问题。如果 L5 仍然失败 → 生成「深层认识论失败报告」并标记为人类研究课题。

### L5 可验证输出

| 产出物 | 格式 |
|---------|------|
| `escalation-log.tsv` 记录 `esc_level=L5` | 包含并行推理路径 + Adversary 攻击报告摘要 |
| 3 份 `reasoning-relay-L5-{A,B,C}.md` | 三路并行推理的独立快照 |
| `adversary-report-L5.md` | Adversary Agent 的完整攻击报告 |
| `tacit-tradition-map.md` 更新 | Polanyi 挖掘的新发现写入 |

## 体面的退出

七项检查清单**全部完成**且仍未解决时，输出结构化的失败报告：

```markdown
## 结构化失败报告

### 已验证事实
[列出已确认的事实，包括验证方式]

### 已排除可能
[列出已排除的假设及排除证据]

### 缩小的范围
[问题最可能在什么范围内]

### 推荐下一步
[推荐人类介入的具体方向]

### 隐性边界警示 (Tacit Boundary Warning) [Polanyi 移交]
[科学共和国移交：提炼原有代码认知认知中的哪个**“隐式假设/潜规则”**被本次排查证伪，或者暴露了哪种“环境深层裂痕”。不留毫无营养的错误堆砌，为下一个接手者提供认识论上的反证起点。]

### 交接信息
[关键文件路径、错误日志位置、环境信息]
```

> 这不是"我不行"。这是"问题的边界在这里"。有尊严的退出。

## Gotchas（已知 AI 行为陷阱）

1. **假装换了方案**：L2 要求"本质不同的方案"，但实际只换了参数/换了个函数名 — 必须检测自己是否真的换了思路
2. **声称穷尽但只试了 2 种**：说"已尝试所有方法"时列出完整清单 — 少于 3 种说明没穷尽
3. **标记和行为脱节**：嘴上说"闭环"但没跑 build，输出了报告但验证列是空的
4. **Sub-agent 裸奔**：spawn 子 agent 时忘了在 prompt 里注入压力升级规则 — 子 agent 是空白上下文

## 可验证输出要求（Verifiable Output）

> ⚠️ 每个压力等级必须产出**可验证的文件输出**，杜绝"嘴上说做了但没有证据"。

| 等级 | 必须产出的可验证输出 |
|------|---------------------|
| **L1** | 在 results TSV 中 `esc_level=L1` + 新方案描述（与前一方案**本质不同**，描述中可对比） |
| **L2** | 在 `escalation-log.tsv` 中记录：搜索关键词、源码上下文位置（file:line）、3 个新假设列表 |
| **L3** | 在 `escalation-log.tsv` 中记录 7 项清单，每项标注 ✅/❌ + 具体动作和输出证据 |
| **L4** | 在 `escalation-log.tsv` 中记录方法论切换事件（from→to + 原因）；如退出则产出结构化失败报告文件 |

**escalation-log.tsv 标准格式**（由 `/autoresearch:debug`、`/autoresearch:fix` 共用）：
```tsv
iteration	level	target	trigger	checklist_completed	methodology_from	methodology_to	details	outcome
```

**L3 清单记录格式**（details 字段）：
```
1.逐字读完失败信号:✅ error.log:42;2.搜索核心问题:✅ grep "XXX" 3 results;3.读原始上下文:✅ auth.ts:30-80;4.假设用工具确认:✅ node -e verified;5.反转假设:✅ tested opposite;6.最小复现:✅ test_repro.ts;7.换工具/方法:✅ switched to data-driven
```

## 反馈闭环协议（Feedback Loop）

> 压力升级系统必须形成**完整的监控 → 检测 → 升级 → 验证 → 记录**闭环。

### 闭环五步

```
① 监控 — PostToolUse 钩子检测命令失败，更新连续失败计数
② 检测 — 失败计数达到阈值时，自动判定压力等级
③ 升级 — 按 L1-L4 执行对应强制动作
④ 验证 — 强制动作的输出必须写入 escalation-log.tsv
⑤ 记录 — results TSV、escalation-log.tsv、summary.md 三处同步
```

### 反馈信号

| 信号 | 含义 | 后续动作 |
|------|------|---------|
| keep 后 esc_level 重置 L0 | 升级策略有效 | 在 escalation-log 记录 `outcome=resolved` |
| L3 清单完成后仍失败 | 问题超出当前能力 | 强制 L4 方法论切换 |
| L4 方法论切换后 keep | 方法论路由有效 | 记录成功的方法论供学习 |
| 全循环 peak ≤ L1 | 任务难度低 | 无需特殊处理 |
| 全循环多次 L3+ | 任务难度高 | summary 建议架构审查 |

### Post-session 经验提取

循环结束后，从 escalation-log.tsv 中提取：
1. **有效 pattern**：哪些方法论切换成功解决了问题？
2. **无效 pattern**：哪些假设反复被排除？
3. **改进建议**：如果同类问题频繁触发 L3+，建议新增 captured skill

> 此提取与 `/learn` 工作流联动 — escalation 经验自动成为持续学习的输入。

| 现有流程 | Escalation 的增强 |
|---------|--------------------|

## 阶段感知调节 (Phase-Aware Adaptation)

> 灵感来源：OpenMythos LoRA Depth Adapter — 共享权重在不同循环深度自动微调行为。
> 核心理念：同一套 escalation 规则在迭代的不同阶段，自动调节**严格程度**。

### 三阶段行为微调

| 阶段 | 迭代范围 | L1 动作 | L2 动作 | 容忍度 | 类比 OpenMythos |
|------|---------|---------|---------|--------|----------------|
| **早期** (Warm-up) | iter 1-3 | 建议换方案（温和提示） | 搜索 + 2 个假设 | 高 — 允许试错 | `loop_t=0-2` |
| **中期** (Cruising) | iter 4-8 | **强制**换方案 | 搜索 + 读源码 + 3 个假设 | 中 — 标准约束 | `loop_t=3-7` |
| **晚期** (Convergence) | iter 9+ | 强制换方案 + 搜索外部知识 | 强制方法论切换 | 低 — 零容忍 | `loop_t=8+` |

### 判定规则

```
current_phase = 
    if iteration <= 3:  "early"
    elif iteration <= 8: "mid"
    else:               "late"

# 晚期阶段强化：
if current_phase == "late":
    L1 动作自动升级为 L2 级别
    L2 动作自动升级为 L3 级别（七项清单）
    同一思路微调 2 次（而非 3 次）即判定"原地打转"
```

### 与 ACT 停机的协同

阶段感知与 ACT 自适应停机互补：
- **ACT** 控制"何时收敛退出"（子任务级别）
- **阶段感知** 控制"失败时多严格"（循环全局级别）
- 两者独立运作，互不干扰
