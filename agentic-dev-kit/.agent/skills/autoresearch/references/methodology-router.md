# 方法论智能路由（Methodology Router）

> 灵感来源：[tanweai/pua v3](https://github.com/tanweai/pua) 的方法论路由系统。
> 核心理念：根据任务类型自动选择最佳问题解决方法论，失败时按预定义链条切换，杜绝原地打转。

## 方法论选择矩阵

根据当前任务类型，自动选择最匹配的方法论：

| 任务类型 | 方法论 | 核心步骤 | 典型场景 |
|---------|--------|---------|---------|
| 🔴 **调试/排错** | RCA 根因分析 | 现场保护 → 信号收集 → 假设树 → 逐一验证 → 根因确认 | Bug 修复、异常排查、性能问题 |
| ⬛ **构建/实现** | 质疑→删除→简化→加速→自动化 | ①质疑需求本身 ②删除不必要部分 ③简化实现 ④加速执行 ⑤自动化重复 | 新功能开发、原型构建 |
| 🔍 **调研/探索** | 搜索优先 | 先搜索 → 多源交叉验证 → 构建信息图谱 → 得出结论 | 技术选型、方案调研、知识获取 |
| 🔶 **架构设计** | Working Backwards | 定义最终用户体验 → 反推所需组件 → 验证技术可行性 | 系统设计、架构重构 |
| 🟡 **性能优化** | 数据驱动 | 建立基线 → 量化瓶颈 → A/B 对照实验 → 只保留有数据支持的改进 | 性能调优、资源优化 |
| 🟠 **默认闭环** | 闭环交付 | 接收 → 确认理解 → 执行 → 验证 → 交付证据 → 复盘 | 通用任务执行 |

## 失败模式检测 → 方法论切换

当检测到以下失败模式时，按切换链依次尝试**不同方法论**（已失败的不重复）：

| 失败模式 | 检测信号 | 切换链（按序尝试） |
|---------|---------|----------------|
| 🔄 **原地打转** | 反复改参数不改思路、3+ 次微调同一处 | 质疑→删除→简化 → 砍中间环节 → RCA 根因分析 |
| 🚪 **放弃/推锅** | "建议手动"、"超出范围"、"请用户检查" | 评估方案值不值得保留 → 集中资源重攻 → 极限简化 |
| 💩 **质量差** | 表面完成实质敷衍、没跑验证就说完了 | 像素级审查 → 极致专注单点做好 → 淘汰不达标部分 |
| 🔍 **没搜就猜** | 凭记忆下结论、不查文档不验证 | 搜索优先 → 深挖源码细节 → 数据驱动验证 |
| ⏸️ **被动等待** | 修完就停等指示、不主动推进 | 只看结果 → 过程透明化 → Owner 意识主动出击 |
| ✅ **空口完成** | 没运行验证命令就声称完成 | 数据验证 → 只认结果 → 闭环交付 |

### 切换前三问（防止无效切换）

**在切换方法论前，必须自检：**

1. 当前方法论的核心步骤都走了吗？（没走完 = 加压力不换方法）
2. 失败是方法论不对还是执行不到位？（执行问题 = 不换方法）
3. 新方法论能解决当前失败模式吗？（不能 = 别切）

## 方法论切换协议

切换时输出标记 `[方法论切换 🔄]` 并说明：
```
[方法论切换 🔄] 从 "RCA 根因分析" → "搜索优先"
原因：当前方法依赖假设推导，但假设已穷尽；切换到搜索优先，从外部数据中寻找线索。
```

## 与 Escalation 的整合

方法论路由是 Escalation 压力系统的**行动指导层**：

| Escalation 等级 | 方法论路由行为 |
|----------------|-------------|
| L1 切换方案 | 保持当前方法论，切换具体方案 |
| L2 深度调查 | **建议**切换方法论（根据失败模式推荐） |
| L3 强制清单 | 继续当前方法论，但清单步骤必须全部走完 |
| L4 最后手段 | **强制**切换方法论（从切换链中选下一个） |

## 与 autoresearch 的整合

在 `/autoresearch:debug` 和 `/autoresearch:fix` 循环中：

1. **初始选择**：根据任务类型自动选择方法论
2. **失败检测**：连续 2 次 discard 后评估失败模式
3. **方法论切换**：按切换链切换，输出 `[方法论切换 🔄]` 标记
4. **切换记录**：在 results log 中记录方法论切换事件
5. **不回头**：已使用过的方法论不再重复尝试
