# 代码审查标准 (Code Review Standards)

> 本文件定义代码审查的标准维度和清单。由 `/review` 工作流和 PR 审查自动引用。
> 审查时按以下 5 个维度逐一检查，每个维度独立评分通过/不通过。

## 审查维度

### 1. 安全性 (Security)
- [ ] 无硬编码密钥、token、密码
- [ ] 所有外部输入已验证和清洗
- [ ] 敏感数据未出现在日志中
- [ ] 权限遵循最小权限原则
- [ ] 无已知的注入风险（SQL、XSS、命令注入）

### 2. 正确性 (Correctness)
- [ ] 核心逻辑与需求/契约一致
- [ ] 边界条件已处理（空值、零值、溢出、并发）
- [ ] 错误处理完整（不吞异常、错误信息含上下文）
- [ ] 无明显的性能退化（N+1 查询、内存泄漏、无限循环）

### 3. 可维护性 (Maintainability)
- [ ] 命名清晰表达意图（变量、函数、类）
- [ ] 函数职责单一，长度合理（≤50 行）
- [ ] 注释说明"为什么"而非"做什么"
- [ ] 无魔法数字，常量已命名
- [ ] 无重复代码（DRY）

### 4. 向后兼容性 (Backward Compatibility)
- [ ] API/接口变更保持向后兼容（除非明确要求 breaking change）
- [ ] 数据库 schema 变更有迁移脚本
- [ ] 配置变更有合理默认值
- [ ] 已有测试未被破坏

### 5. 测试覆盖 (Test Coverage)
- [ ] 新增/修改的代码有对应测试
- [ ] 测试覆盖正常路径 + 至少一个边界/异常路径
- [ ] 测试命名清晰：`test_<功能>_<场景>_<预期结果>`
- [ ] 全量测试通过

## 审查输出格式

```markdown
## 代码审查报告
- **审查范围**: [文件/模块列表]

### 各维度结果
| 维度 | 状态 | 备注 |
|---|---|---|
| 安全性 | ✅/❌ | [问题说明] |
| 正确性 | ✅/❌ | [问题说明] |
| 可维护性 | ✅/❌ | [问题说明] |
| 向后兼容性 | ✅/❌ | [问题说明] |
| 测试覆盖 | ✅/❌ | [问题说明] |

### 已确认问题
- [问题] — 严重等级: [等级] — 建议修复: [方案]

### 建议改进（非阻塞）
- [建议] — 理由: [说明]
```
