# 测试规范规则

> 当 Agent 操作测试文件或编写测试时，本规则自动生效。

## 测试命名
- [ ] 格式：`test_<功能>_<场景>_<预期结果>`
- [ ] 描述必须清晰表达测试意图，无需阅读代码即可理解

## 测试设计
- [ ] 每个测试只验证一个行为（Single Assertion Principle）
- [ ] 测试之间相互独立，不依赖执行顺序
- [ ] 每个测试必须包含：正常路径 + 至少一个边界条件 + 至少一个异常路径
- [ ] 使用 mock/stub 隔离外部依赖（数据库、API、文件系统）

## 禁止事项
- [ ] ❌ 禁止在测试中使用 `sleep` 或硬编码延迟
- [ ] ❌ 禁止修改测试来通过验收（除非测试本身设计有误且经人类确认）
- [ ] ❌ 禁止凭记忆推断测试结果，必须以实际命令输出为准
- [ ] ❌ 禁止跳过失败的测试而不修复

## 执行纪律
- [ ] 修改代码后必须运行相关测试再提交
- [ ] 全量测试通过是工作流闭环的前置条件
- [ ] 优先运行单个测试文件而非整个测试套件（提升速度）

## TDD 模式规则（`/tdd` 工作流激活时强制生效）
- [ ] 🔴 必须先写一个失败测试并运行确认失败，才可编写实现代码
- [ ] 🟢 实现代码必须是让测试通过的**最小代码**，不可超前编写测试未要求的功能
- [ ] 🔵 重构必须在全量测试绿灯下进行，不可改变外部行为
- [ ] ❌ 禁止跳过红灯步骤（不运行测试就写实现 = 不是 TDD）
- [ ] ❌ 禁止一次性写完所有测试再实现（应逐个循环推进）
- [ ] ❌ 禁止编写依赖实现内部细节的脆弱测试（测试行为，不测试实现）


## 自动化合规验证
可以使用如下命令验证当前环境状态：
```bash
bash .agent/scripts/health-check.sh .
```
